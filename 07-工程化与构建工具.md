## 7.0 Vue CLI 脚手架

**Vue CLI 是 Vue.js 官方提供的脚手架工具：**

1. **项目脚手架**
   - 创建 Vue 项目基础结构
   - 选择配置（TypeScript、CSS 预处理器等）

2. **开发服务器**
   - 内置 webpack-dev-server
   - 热模块替换（HMR）
   - 自动打开浏览器

3. **构建打包**
   - 代码打包、压缩、优化
   - Code Splitting（代码分割）
   - Tree Shaking（清除未使用代码）

4. **加载器配置**
   - babel-loader（JS 语法转换）
   - css-loader / sass-loader
   - file-loader / url-loader

5. **插件系统**
   - 内置插件：HtmlWebpackPlugin、MiniCssExtractPlugin
   - 支持自定义插件

6. **图形化配置**
   - `vue ui` 命令打开图形界面
   - 可视化配置 webpack

**Vue CLI vs Vite：**

| 特性 | Vue CLI | Vite |
|------|---------|------|
| 构建速度 | 较慢（全量构建） | 快（按需加载） |
| 开发服务器 | webpack-dev-server | 原生 ESM |
| HMR | 较慢 | 极快 |
| 生产构建 | webpack | Rollup |

**题目：Vue CLI 提供了哪些功能？它和 Vite 有什么区别？**

**满分答案：**

**Vue CLI 功能：**
1. 项目脚手架生成
2. 开发服务器和热更新
3. 生产环境构建打包
4. 插件和加载器配置
5. 图形化配置界面
6. 单元测试和 E2E 测试支持

**Vue CLI vs Vite：**
- Vite 是新一代构建工具，更快
- Vue CLI 基于 webpack，更成熟
- 新项目推荐 Vite，老项目继续用 Vue CLI

## 7.1 Webpack

### 7.1.1 核心概念

#### 知识点详解

Webpack 是一个现代 JavaScript 应用程序的静态模块打包工具。

**核心概念：**

1. **Entry（入口）**：Webpack 开始构建的起点
2. **Output（输出）**：打包后文件的输出位置
3. **Loader**：处理非 JavaScript 文件（CSS、图片等）
4. **Plugin**：扩展 Webpack 功能
5. **Mode**：开发/生产模式

**基础配置：**

```javascript
// webpack.config.js
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');

module.exports = {
    // 入口
    entry: {
        main: './src/index.js',
        vendor: './src/vendor.js'
    },
    
    // 输出
    output: {
        path: path.resolve(__dirname, 'dist'),
        filename: '[name].[contenthash].js',
        chunkFilename: '[name].[contenthash].chunk.js',
        clean: true  // 每次构建前清理 dist
    },
    
    // 模式
    mode: 'production',  // 'development' | 'production' | 'none'
    
    // Source Map
    devtool: 'source-map',
    
    // 模块解析
    resolve: {
        extensions: ['.js', '.jsx', '.ts', '.tsx'],
        alias: {
            '@': path.resolve(__dirname, 'src')
        }
    },
    
    // Loader
    module: {
        rules: [
            // JavaScript/TypeScript
            {
                test: /\.(js|jsx|ts|tsx)$/,
                exclude: /node_modules/,
                use: {
                    loader: 'babel-loader',
                    options: {
                        presets: [
                            '@babel/preset-env',
                            '@babel/preset-react',
                            '@babel/preset-typescript'
                        ]
                    }
                }
            },
            
            // CSS
            {
                test: /\.css$/,
                use: [
                    MiniCssExtractPlugin.loader,  // 生产环境提取 CSS
                    // 'style-loader',            // 开发环境注入 style 标签
                    'css-loader',
                    'postcss-loader'
                ]
            },
            
            // SCSS
            {
                test: /\.scss$/,
                use: [
                    MiniCssExtractPlugin.loader,
                    'css-loader',
                    'postcss-loader',
                    'sass-loader'
                ]
            },
            
            // 图片
            {
                test: /\.(png|jpg|gif|svg|webp)$/,
                type: 'asset',
                parser: {
                    dataUrlCondition: {
                        maxSize: 8 * 1024  // 8KB 以下转 base64
                    }
                },
                generator: {
                    filename: 'images/[name].[hash][ext]'
                }
            },
            
            // 字体
            {
                test: /\.(woff|woff2|eot|ttf|otf)$/,
                type: 'asset/resource',
                generator: {
                    filename: 'fonts/[name].[hash][ext]'
                }
            }
        ]
    },
    
    // Plugin
    plugins: [
        new HtmlWebpackPlugin({
            template: './public/index.html',
            filename: 'index.html',
            minify: {
                removeComments: true,
                collapseWhitespace: true
            }
        }),
        new MiniCssExtractPlugin({
            filename: 'css/[name].[contenthash].css'
        })
    ],
    
    // 优化
    optimization: {
        splitChunks: {
            chunks: 'all',
            cacheGroups: {
                vendor: {
                    test: /[\\/]node_modules[\\/]/,
                    name: 'vendors',
                    chunks: 'all'
                }
            }
        },
        runtimeChunk: 'single'
    },
    
    // 开发服务器
    devServer: {
        port: 3000,
        hot: true,
        historyApiFallback: true,
        proxy: {
            '/api': {
                target: 'http://localhost:8080',
                changeOrigin: true
            }
        }
    }
};
```

### 7.1.3 自定义 Plugin

```javascript
// MyPlugin.js
class MyPlugin {
    constructor(options) {
        this.options = options;
    }
    
    apply(compiler) {
        // 在 emit 钩子（输出文件前）执行
        compiler.hooks.emit.tapAsync('MyPlugin', (compilation, callback) => {
            // 获取所有输出文件
            const assets = compilation.assets;
            
            // 创建一个文件清单
            let manifest = '';
            for (const filename in assets) {
                manifest += `${filename}\n`;
            }
            
            // 添加到输出
            compilation.assets['manifest.txt'] = {
                source: () => manifest,
                size: () => manifest.length
            };
            
            callback();
        });
    }
}

module.exports = MyPlugin;
```

## 7.2 Vite

### 7.2.1 Vite 原理

**为什么 Vite 快：**

1. **开发环境**：
   - 利用浏览器原生 ES Modules
   - 按需编译，不打包
   - 使用 esbuild 预构建依赖（比 Babel 快 10-100 倍）

2. **生产环境**：
   - 使用 Rollup 打包
   - 更好的 Tree Shaking

```javascript
// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import vue from '@vitejs/plugin-vue';
import path from 'path';

export default defineConfig({
    plugins: [
        react(),
        // vue()
    ],
    
    resolve: {
        alias: {
            '@': path.resolve(__dirname, './src')
        }
    },
    
    server: {
        port: 3000,
        proxy: {
            '/api': {
                target: 'http://localhost:8080',
                changeOrigin: true,
                rewrite: (path) => path.replace(/^\/api/, '')
            }
        }
    },
    
    build: {
        outDir: 'dist',
        sourcemap: true,
        rollupOptions: {
            output: {
                manualChunks: {
                    vendor: ['react', 'react-dom'],
                    router: ['react-router-dom']
                }
            }
        }
    },
    
    css: {
        preprocessorOptions: {
            scss: {
                additionalData: `@import "@/styles/variables.scss";`
            }
        }
    }
});
```

### 7.2.2 Vite 深入：为什么比 Webpack 快

#### 知识点详解

**Webpack 构建流程：**

```
1. 读取配置文件
2. 递归分析依赖（AST）
3. 打包成一个/多个 bundle
4. 代码转换（Babel/TypeScript）
5. 代码压缩/优化
6. 输出到 dist

问题：每次修改都要重新打包整个项目
```

**Vite 构建原理：**

```
开发环境：
1. 启动开发服务器
2. 不打包！直接用原生 ESM
3. 浏览器请求哪个模块，Vite 就编译哪个
4. 依赖用 esbuild 预构建（.js → .esm.js）

生产环境：
1. 用 Rollup 打包（比 Webpack 快）
2. Tree Shaking 更彻底
3. 代码分割更智能
```

**Vite 快的核心原因：**

```javascript
// 1. ESM 按需加载（无需打包）
// 浏览器原生支持
import { add } from './math.js';  // 直接请求这个文件

// Vite 拦截请求，现场编译
// 返回：export function add() {}

// 2. esbuild 预构建（极快）
// 第一次启动时，把 node_modules 里的 CommonJS 转成 ESM
// esbuild 是 Go 写的，比 Babel 快 10-100 倍

// 3. 依赖预构建
// import { Button } from 'antd'
// ↓ 预构建后
// import Button from '/node_modules/.vite/antd/esm/Button.js'

// 4. 热更新快
// Webpack: 修改 → 重新打包 → 页面刷新
// Vite: 修改 → 只更新那个模块 → HRM 局部更新
```

**Vite vs Webpack 对比：**

| 特性 | Webpack | Vite |
|------|---------|------|
| 开发启动 | 慢（需打包） | 快（不打包） |
| 热更新 | 较慢 | 极快 |
| 构建速度 | 慢 | 快（Rollup） |
| 依赖预构建 | 无 | esbuild |
| ESM 支持 | 需要转换 | 原生支持 |
| 生态 | 成熟丰富 | 较新 |

**适用场景：**

```javascript
// 适合 Vite
- Vue 3 项目
- React 项目（用 @vitejs/plugin-react）
- 快速原型开发
- 小型项目

// 仍需 Webpack
- 复杂的大型项目
- 需要大量自定义配置
- 老项目迁移
- 特殊加载器需求
```

#### 真实面试题

**题目：Vite 为什么比 Webpack 快？它的构建原理是什么？**

**满分答案：**

**开发环境快的原因：**
1. **不打包**：利用浏览器原生 ESM，按需编译
2. **esbuild 预构建**：Go 语言写的编译器，比 Babel 快 10-100 倍
3. **HMR 局部更新**：只更新变化的模块，不重整个页面

**生产环境快的原因：**
1. **Rollup 打包**：比 Webpack 更适合库打包
2. **更好的 Tree Shaking**：基于 ESM 分析

**核心区别：**
- Webpack：先打包再运行
- Vite：先运行，按需编译

### 7.3.1 ESLint

**ESLint 检查过程：**

```
1. 解析 → 2. 规则匹配 → 3. 报告错误 → 4. 修复/阻止提交
```

**1. 解析阶段：**

```javascript
// ESLint 使用 Espree 解析代码
const espree = require('espree');
const ast = espree.parse(code, {
    ecmaVersion: 2020,
    sourceType: 'module'
});
```

**2. 规则匹配：**

```javascript
// 规则结构
const rule = {
    meta: {
        type: 'problem',
        docs: { description: 'disallow var' }
    },
    create(context) {
        return {
            VariableDeclaration(node) {
                if (node.kind === 'var') {
                    context.report({
                        node,
                        message: 'Use let or const instead of var'
                    });
                }
            }
        };
    }
};
```

#### 真实面试题

**题目：ESLint 的检查过程是什么？它是如何发现代码问题的？**

**满分答案：**

**ESLint 检查过程：**

1. **解析代码**：将 JS 代码解析为 AST（抽象语法树）
2. **加载配置**：读取 .eslintrc.js 等配置文件
3. **加载规则**：根据配置加载对应规则
4. **遍历 AST**：使用 visitor 模式遍历语法树
5. **规则匹配**：每个规则检查对应节点
6. **报告错误**：发现问题使用 context.report 报告
7. **自动修复**：使用 --fix 自动修复可修复的问题

**集成方式：**
- VS Code 保存时自动检查
- Git Hook 提交前检查
- CI/CD 流水线检查

#### 真实面试题

**题目：Webpack 和 Vite 有什么区别？为什么 Vite 开发环境更快？**

**满分答案：**

**Webpack 的工作方式：**

1. 从入口文件开始，递归分析所有依赖
2. 将所有模块打包成一个或多个 bundle
3. 开发服务器启动时需要完整打包
4. 每次修改都需要重新打包（虽然有 HMR 优化）

**Vite 的工作方式：**

1. **开发环境**：
   - 利用浏览器原生 ES Modules
   - 不打包，按需编译
   - 使用 esbuild 预构建 node_modules（比 Babel 快 10-100 倍）
   - 修改文件时只需重新编译该文件

2. **生产环境**：
   - 使用 Rollup 打包
   - 更好的 Tree Shaking

**性能对比：**

| 指标 | Webpack | Vite |
|------|---------|------|
| 冷启动 | 慢（需要打包） | 快（按需编译） |
| HMR | 较慢 | 极快 |
| 构建速度 | 较慢 | 快（esbuild） |
| 生产构建 | 成熟稳定 | 较新 |
| 生态 | 非常丰富 | 快速增长 |

**为什么 Vite 开发环境快：**

```
Webpack 开发流程：
源码 → 打包 → bundle.js → 浏览器

Vite 开发流程：
源码 → 浏览器请求 → 按需编译 → 返回 ES Module
```

1. **无需打包**：浏览器直接加载 ES Modules
2. **esbuild 预构建**：Go 语言编写，比 JS 快 10-100 倍
3. **按需编译**：只编译当前页面需要的文件
4. **精准 HMR**：只更新修改的模块，不影响其他模块

**选择建议：**

- 新项目：优先选择 Vite
- 已有 Webpack 项目：评估迁移成本
- 需要复杂自定义配置：Webpack 更成熟
- 追求开发体验：Vite 更好

### 7.x.x package.json 中 sideEffects 属性的作用 [必会]

#### 知识点详解

`sideEffects` 是 Webpack 4+ 新增的配置项，用于告诉 Webpack 哪些文件是"纯函数"（无副作用），从而支持 **Tree Shaking** 优化。

**基本用法：**

```json
// package.json
{
  "sideEffects": [
    "*.css",
    "*.less",
    "./src/some-module.js"
  ]
}
```

**两种配置方式：**

| 配置 | 含义 |
|------|------|
| `sideEffects: false` | 所有文件都是纯函数，可以安全地 Tree Shaking |
| `sideEffects: [...]` | 明确标记哪些文件有副作用（保留），其他文件可摇树 |

**实际例子：**

```js
// math.js — 纯函数，无副作用
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }

// ui.js — 有副作用（注册全局组件）
export const Button = () => {...};
window.MyUI = { Button }; // ← 副作用！这行代码不能删除

// index.js
export { add, multiply } from './math';  // 可被 Tree Shaking
export { Button } from './ui';            // ui.js 整体被保留（因为有副作用）
```

```json
{
  "sideEffects": ["./src/ui.js", "*.css"]
}
```

**为什么 CSS 也需要标记？**

```js
// 如果不标记 *.css 为有副作用，Webpack 会认为可以安全删除
import './style.css';
// 如果没有 "*.css" 在 sideEffects 中，这行导入会被 Tree Shaking 删掉！
```

**Tree Shaking 的工作原理：**

```
源代码          标记后           Tree Shaking 后
add(a,b)     →  add(a,b)       →  add(a,b)     (保留)
multiply()   →  multiply()      →  (删除)       (未使用)
Button       →  Button (有副作用) → Button       (保留)
```

**常见面试题：**

> **Q：为什么 CSS 文件要加入 sideEffects？**
> 因为 CSS 文件本身不导出任何内容，只通过 `@import` 或 `import` 被引入。如果不标记，Webpack 会认为它"没用"而删除掉。

> **Q：sideEffects 和 externals 有什么区别？**
> - `sideEffects`：优化产物大小，告诉 Webpack 哪些文件可以安全删除
> - `externals`：告诉 Webpack 不打包某些库（从外部 CDN/Node_modules 加载）

#### 面试加分项
- 了解 Tree Shaking 的前提：必须是 ES Module（静态 import/export），CommonJS 不支持
- 知道在 package.json 设置 `"sideEffects": false` 意味着所有模块都是纯函数

## 7.7 Git pull 和 Git fetch 区别

#### 知识点详解

**Git Fetch：**

```bash
# 获取远程更新，但不合并
git fetch origin

# 获取所有分支
git fetch --all

# 获取并查看差异
git fetch -v
```

**Git Pull：**

```bash
# 获取并合并远程更新
git pull origin main

# 相当于 git fetch + git merge
git pull = git fetch + git merge

# 使用 rebase
git pull --rebase origin main
```

**核心区别图解：**

```
git fetch:
┌─────────────┐     ┌─────────────┐
│   Remote    │ ──→ │   Local     │
│  origin/main│     │  origin/main│  (更新了，但本地 main 没变)
└─────────────┘     └─────────────┘

git pull:
┌─────────────┐     ┌─────────────┐
│   Remote    │ ──→ │   Local     │
│  origin/main│     │  origin/main│ ──→ main (自动合并)
└─────────────┘     └─────────────┘
```

**最佳实践：**

```bash
# 1. 先看差异
git fetch origin
git log HEAD..origin/main --oneline

# 2. 确认安全后再合并
git merge origin/main

# 3. 或者使用 rebase 保持线性历史
git pull --rebase origin main
```

#### 真实面试题

**题目：git pull 和 git fetch 有什么区别？**

**满分答案：**

| 操作 | git fetch | git pull |
|------|----------|---------|
| 获取远程更新 | ✅ | ✅ |
| 自动合并 | ❌ | ✅ |
| 本地分支更新 | ❌ | ✅ |
| 安全性 | 高（可先查看） | 中（直接合并） |

**推荐工作流：**
1. `git fetch` 获取远程更新
2. `git diff` 查看差异
3. 确认无误后 `git merge` 或 `git pull`

## 7.9 CI/CD 核心概念详解

### 7.9.1 CI、CD、持续交付与持续部署的区别

#### 知识点详解

**持续集成（CI — Continuous Integration）：**

```
开发者提交代码 → 自动触发构建 → 运行测试 → 快速反馈
```

**核心目标：**
- 每次提交都触发自动化构建和测试
- 尽早发现集成错误
- 确保代码库始终处于可工作状态

**持续交付（CD — Continuous Delivery）：**

```
CI 完成 → 自动部署到 staging 环境 → 人工确认 → 手动触发生产部署
```

**核心目标：**
- 代码始终处于可部署状态
- 通过人工审批控制何时正式发布
- 适合对发布节奏有严格控制的企业

**持续部署（CD — Continuous Deployment）：**

```
CI 完成 → 自动部署到 staging → 自动部署到生产环境
```

**核心目标：**
- 代码通过所有测试后**零人工干预**直接进入生产
- 最大化发布频率
- 适合 DevOps 成熟度高的团队

**三者关系图解：**

```
CI（持续集成）: 每次提交 → 构建 + 测试
    ↓
CD（持续交付）: 构建 + 测试 + staging 可用 + 人工审批
    ↓
CD（持续部署）: 构建 + 测试 + 自动上线（无需人工）
```

**对比表格：**

| 维度 | CI | CD（持续交付） | CD（持续部署） |
|------|-----|-------------|-------------|
| 自动化构建 | ✅ | ✅ | ✅ |
| 自动化测试 | ✅ | ✅ | ✅ |
| 自动化到 staging | ✅ | ✅ | ✅ |
| 自动化到生产 | ❌ | ❌（需人工） | ✅ |
| 人工审批 | — | ✅ | ❌ |
| 发布速度 | 快 | 中等 | 最快 |
| 风险控制 | 较好 | 好 | 需配套灰度 |

**CI 的关键实践：**

```yaml
# GitHub Actions 示例：CI 核心流程
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 1. 依赖安装（利用缓存加速）
      - uses: actions/cache@v3
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}

      - run: npm ci

      # 2. Lint 检查
      - run: npm run lint

      # 3. 单元测试
      - run: npm test

      # 4. 构建
      - run: npm run build

      # 5. 推送镜像
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: registry.example.com/app:${{ github.sha }}
```

**DevOps 核心原则：**

1. **自动化**：所有重复性操作（构建、测试、部署）自动化
2. **协作**：打破开发与运维之间的壁垒
3. **快速反馈**：缩短从提交到发现问题的时间
4. **持续改进**：基于监控数据不断优化流程
5. **基础设施即代码（IaC）**：用代码管理服务器、网络等基础设施

#### 真实面试题

**题目：解释 CI、CD、持续交付和持续部署的区别，以及它们之间的关系。**

**满分答案：**

**CI（持续集成）：**
- 开发者频繁将代码合并到共享主干（每天多次）
- 每次合并自动触发构建 + 自动化测试
- 目标：尽早发现集成错误，确保代码库始终可工作

**CD（持续交付 — Continuous Delivery）：**
- 在 CI 基础上，确保代码**始终处于可部署状态**
- 自动部署到 staging/UAT 环境
- 生产部署需要**人工审批确认**
- 适合对发布节奏有严格控制的企业

**CD（持续部署 — Continuous Deployment）：**
- 在 CI 基础上，所有环境（包括生产）**全自动化**
- 代码通过所有测试后无需人工干预直接上线
- 最大化发布频率，适合 DevOps 成熟团队

**三者关系：**
```
CI ⊂ CD(持续交付) ⊂ CD(持续部署)
CI 是基础，CD(持续交付)是扩展，持续部署是最高自动化等级
```

**选择建议：**
- 金融/核心系统：优先 CI + CD（持续交付），人工审批保安全
- 互联网产品：CI + CD（持续部署），配合灰度发布降低风险

### 7.9.2 CI/CD Pipeline 完整设计

#### 知识点详解

**CI/CD Pipeline 典型阶段：**

```
代码提交 → Git Hook 触发
    ↓
1. 检出代码（Checkout）
    ↓
2. 静态检查（Lint / SonarQube）
    ↓
3. 安装依赖（npm ci / yarn）
    ↓
4. 单元测试（jest / vitest）
    ↓
5. 集成测试
    ↓
6. 安全扫描（SAST / 依赖漏洞扫描）
    ↓
7. 构建产物（Build）
    ↓
8. 镜像构建（Docker Build & Push）
    ↓
9. 部署到测试环境
    ↓
10. E2E / 自动化验收测试
    ↓
11. 部署到预生产环境
    ↓
12. 生产部署（手动/自动）
    ↓
13. 监控 & 告警
```

11. **部署预生产**：人工审批后部署

### 7.9.3 部署策略详解：蓝绿、金丝雀、滚动发布

#### 知识点详解

**三种部署策略对比：**

```
┌──────────────────────────────────────────────────────────────┐
│                     蓝绿部署（Blue-Green）                    │
│  ┌──────────────────┐      ┌──────────────────┐              │
│  │   Blue（旧版本）   │ ←切→ │   Green（新版本） │              │
│  │   接收100%流量    │      │   准备上线        │              │
│  └──────────────────┘      └──────────────────┘              │
│  切换方式：负载均衡器瞬间切换，全量替换                       │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                    金丝雀部署（Canary）                       │
│  ┌──────────────────┐      ┌──────────────────┐              │
│  │   v1（旧版本）     │ 90% │  v2（新版本）      │ 10%         │
│  │                   │ ←→ │                   │              │
│  └──────────────────┘      └──────────────────┘              │
│  切换方式：渐进式流量分配 10% → 30% → 50% → 100%            │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                    滚动发布（Rolling）                       │
│  ┌────┐ ┌────┐ ┌────┐                                       │
│  │ v2 │ │ v1 │ │ v2 │   逐个替换实例，直到全部为 v2          │
│  └────┘ └────┘ └────┘                                       │
└──────────────────────────────────────────────────────────────┘
```

**金丝雀部署 Kubernetes 实现（含 Argo Rollouts）：**

```yaml
# canary-deployment.yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp-rollout
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 5          # 先放 5% 流量
        - pause: {duration: 10m}  # 观察 10 分钟
        - setWeight: 20         # 增加到 20%
        - pause: {duration: 10m}
        - setWeight: 50
        - pause: {}             # 手动确认继续
        - setWeight: 80
        - pause: {duration: 5m}
        - setWeight: 100        # 全量
      canaryMetadata:           # 金丝雀版本标签
        labels:
          role: canary
      stableMetadata:           # 稳定版本标签
        labels:
          role: stable
      trafficRouting:
        nginx:
          stableIngress: myapp-ingress
          additionalIngressAnnotations:
            canary-by-header: X-Canary
      analysis:
        templates:
          - templateName: success-rate
        startingStep: 1
        args:
          - name: service-name
            value: myapp-rollout-canary
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:2.0.0
          ports:
            - containerPort: 8080

---
# 自动分析模板（基于指标自动回滚）
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 1m
      successCondition: result[0] >= 0.95
      failureLimit: 3           # 连续 3 次失败则回滚
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{
              job="{{args.service-name}}",
              status!~"5.*"}[1m]))
            /
            sum(rate(http_requests_total{
              job="{{args.service-name}}"}[1m]))
```

```yaml
# 基于权重的 Ingress 配置
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"  # 10% 到金丝雀

  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: myapp-canary
                port:
                  number: 80

---
# 基于 Header 的金丝雀（内部用户优先体验）
# X-Canary: always → 总是访问新版本
# X-Canary: never  → 总是访问旧版本
nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
nginx.ingress.kubernetes.io/canary-by-header-value: "always"
```

**三种策略适用场景对比：**

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 蓝绿部署 | 切换快、回滚快、零停机 | 资源消耗 200%、要求完全兼容 | 金融核心系统、破坏性更新 |
| 金丝雀部署 | 风险可控、灰度验证、资源省 | 实施复杂、版本共存 | 互联网产品、A/B测试 |
| 滚动发布 | 资源利用率高、自动化 | 回滚慢、部分实例版本不一致 | K8s 默认策略、常规更新 |

#### 真实面试题

**题目：什么是蓝绿部署和金丝雀发布？它们各自适用于什么场景？如何在 Kubernetes 中实现？**

**满分答案：**

**蓝绿部署（Blue-Green Deployment）：**
- 维护两套完全相同的生产环境（蓝＝旧，绿＝新）
- 通过负载均衡器瞬间切换全部流量
- 切换后可快速回滚（切回蓝色即可）
- 缺点：需要双倍资源（200%），新旧版本必须数据库兼容

**金丝雀发布（Canary Deployment）：**
- 先将新版本部署给一小部分用户（如 5%）
- 监控关键指标（错误率、延迟、转化率）
- 指标正常则逐步扩大：5% → 20% → 50% → 100%
- 指标异常自动/手动回滚
- 适合：互联网产品、机器学习模型 A/B 测试

**K8s 实现方式：**

```bash
# 蓝绿：改 Service selector 切换流量
kubectl patch service myapp -p '{"spec":{"selector":{"version":"green"}}}'

# 金丝雀：用 Argo Rollouts 或 Ingress 权重
# Argo Rollouts 示例
kubectl argo rollouts set weight myapp-rollout 10  # 10% 到新版本
kubectl argo rollouts fullPromotion myapp-rollout  # 全量

# 滚动：Deployment 默认 strategy
kubectl rollout status deployment/myapp
```

**选型建议：**
- 金融/核心交易：蓝绿（零容忍错误，全量切换有保障）
- 互联网产品迭代：金丝雀（渐进验证，降低全量风险）
- K8s 常规更新：滚动发布（资源利用率最高）

### 7.9.4 GitOps 详解：ArgoCD 与声明式部署

#### 知识点详解

**GitOps 核心概念：**

```
┌─────────────────────────────────────────────────────────┐
│                      Git 仓库（单一真相来源）              │
│  应用代码 │ Helm Charts │ K8s Manifests │ 参数配置       │
└───────────────┬─────────────────────────────────────────┘
                ↓ 每次 git push 自动触发
        ┌───────────────┐
        │   Argo CD     │  自动同步到集群
        │  监听 Git 变化 │
        └───────┬───────┘
                ↓
        ┌───────────────┐
        │ Kubernetes    │  应用运行在集群中
        │   集群        │
        └───────────────┘
```

**GitOps 四大原则：**

1. **一切皆声明**：所有基础设施和应用配置以代码形式存在
2. **Git 是唯一真相来源**：集群状态由 Git 驱动
3. **自动同步**：Git 变化自动应用到集群
4. **不可变性**：通过 Git 操作而非手动 kubectl

**ArgoCD 安装与基础使用：**

```bash

# 登录
argocd login localhost:8080 --username admin
```

**ArgoCD 应用配置（Application）：**

```yaml

  name: myapp
  namespace: argocd
spec:
  project: default

  # Git 仓库来源
  source:
    repoURL: https://github.com/example/k8s-manifests.git
    targetRevision: main
    path: ./myapp/overlays/prod  # 环境特定路径

    # Helm 方式
    chart: myapp
    helm:
      valueFiles:
        - values-prod.yaml
      parameters:
        - name: image.tag
          value: v2.1.0

  # 目标集群
  destination:
    server: https://kubernetes.default.svc
    namespace: myapp

  # 同步策略
  syncPolicy:
    automated:
      prune: true              # 自动删除 Git 中已移除的资源
      selfHeal: true           # 集群手动修改会自动恢复
      allowEmpty: false
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

  # 忽略差异（如运维侧手动修改）
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
```

#### 真实面试题

**题目：什么是 GitOps？它和传统 CI/CD 部署有什么区别？ArgoCD 是如何工作的？**

**满分答案：**

**GitOps 核心概念：**
- 将 Git 作为"单一真相来源"（Single Source of Truth）
- 所有应用配置和基础设施配置以声明式代码存储在 Git 中
- 自动化工具（如 ArgoCD）监听 Git 变化，自动同步到集群
- "你不需要 SSH 到服务器去改配置，改 Git 就行"

**GitOps vs 传统 CI/CD：**

| 维度 | 传统 CI/CD | GitOps |
|------|----------|--------|
| 部署方向 | Push（CI 推送到集群） | Pull（集群主动拉取） |
| 真相来源 | CI Pipeline 配置 | Git 仓库 |
| 权限需求 | CI 系统需要集群写权限 | 集群只有读权限，更安全 |
| 配置漂移 | 难检测 | 自动检测并修复 |
| 回滚 | 重新执行 Pipeline | `git revert` + 自动同步 |
| 审计 | 依赖 CI 日志 | Git commit 历史天然审计 |

**ArgoCD 工作原理：**

1. **声明式定义**：用户创建 Application，指向 Git 仓库
2. **持续监听**：ArgoCD 持续对比 Git 状态 vs 集群实际状态
3. **自动同步**：发现差异时，自动将集群状态调整为与 Git 一致
4. **多环境支持**：通过 Kustomize/Helm overlays 支持多环境差异化
5. **高级策略**：支持蓝绿、金丝雀（通过 Argo Rollouts）

**GitOps 完整流程：**
```
开发者 → git push → Git 仓库
                        ↓
                 ArgoCD 检测到变化
                        ↓
           ArgoCD 对比 Git vs 集群状态
                        ↓
           差异 → 自动 kubectl apply
                        ↓
           集群状态 = Git 声明的状态 ✓
```

### 7.9.6 CI/CD 安全实践与监控

#### 知识点详解

**CI/CD 安全十大实践：**

```
1. 最小权限原则（Least Privilege）
   - CI 系统只有部署所需权限，不给过多
   - 使用 IAM Roles / Kubernetes RBAC

2. 密钥和凭证管理（Secrets Management）
   - 使用 Vault / AWS Secrets Manager / K8s Secrets
   - 绝对不能在代码或 Pipeline 中硬编码密钥

3. 依赖安全（Supply Chain Security）
   - 依赖漏洞扫描：npm audit / Trivy / Snyk
   - 锁定依赖版本：package-lock.json / Pipfile.lock

4. 镜像安全（Container Security）
   - 使用最小化基础镜像（alpine）
   - 不以 root 用户运行容器
   - 镜像签名和校验

5. 代码签名（Code Signing）
   - commit 签名验证（GPG）
   - 镜像签名（Cosign / Notary）

6. Pipeline 安全性（Pipeline as Code）
   - Pipeline 配置纳入版本控制
   - 定期审计 Pipeline 配置

7. 网络隔离（Network Segmentation）
   - CI Runner 与生产网络隔离
   - 使用私有网络 / VPC

8. 审计日志（Audit Logs）
   - 记录所有部署操作
   - 与 SIEM 系统集成

9. 运行时安全监控
   - Falco / Sysdig 监控容器异常行为
   - 实时告警

10. 灾备与恢复（DR）
    - 定期备份 CI/CD 配置
    - 多区域/多集群容灾
```

**Secrets 安全管理最佳实践：**

```yaml
# ❌ 错误：硬编码密钥在 Pipeline 中
script:
  - echo "SECRET=abc123" >> .env  # 绝对不要这样做！

# ✅ 正确：使用 CI/CD 内置 Secrets
# GitHub Actions
env:
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

# GitLab CI
variables:
  DOCKER_REGISTRY_PASSWORD: $CI_REGISTRY_PASSWORD

# ✅ 正确：使用 Vault 动态凭证
- name: Fetch secrets from Vault
  uses: hashicorp/vault-action@v2
  with:
    url: https://vault.example.com
    method: aws
    role: myapp-deploy
    secrets: |
      secret/data/myapp prod_db_password | DB_PASSWORD;
      secret/data/myapp api_key | API_KEY;
```

**Docker 安全最佳实践：**

```dockerfile

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# ✅ 使用非 root 用户

RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# ✅ 重要文件只读
COPY --from=builder /app/nginx.conf /etc/nginx/nginx.conf
```

### 7.9.7 回滚机制与灾难恢复

#### 知识点详解

**回滚是 CI/CD 的最后一道防线：** 一旦新版本出现问题，能否快速回滚决定了故障影响时长。

**K8s Deployment 回滚：**

```bash
# 查看部署历史
kubectl rollout history deployment/myapp

# 查看特定版本的详情
kubectl rollout history deployment/myapp --revision=3

# 回滚到上一个版本
kubectl rollout undo deployment/myapp

# 回滚到指定版本
kubectl rollout undo deployment/myapp --to-revision=2

# 监听回滚进度
kubectl rollout status deployment/myapp

# 确认回滚成功
kubectl get pods -l app=myapp
kubectl describe deployment myapp | grep Image
```

**ArgoCD 回滚（GitOps 方式）：**

```bash
# 查看历史版本
argocd app history myapp

# 回滚到指定版本（Revision）
argocd app rollback myapp <revision>

# 效果：自动将 Git 指向对应 commit，自动同步到集群
```

**GitOps 回滚的两种方式：**

```bash
# 方式一：git revert（推荐，记录回滚历史）
git checkout main
git revert HEAD              # 创建一个撤销上次的 commit
git push origin main         # ArgoCD 自动同步 → 自动回滚

# 方式二：git reset（线性历史，不推荐在 main 分支）
git checkout main
git reset --hard HEAD~1      # 丢弃上次 commit
git push origin main --force # ⚠️ 危险操作！会覆盖远程历史

# 方式三：直接指定历史版本
git checkout <old-commit-hash>
git tag v1.0.0-rollback
git push origin v1.0.0-rollback
argocd app sync myapp         # ArgoCD 同步到该 tag
```

**数据库迁移的回滚策略：**

```javascript
// 不可逆迁移（ALTER TABLE）→ 必须前后兼容
// 原则：数据库变更在应用变更之前
// 回滚：应用回滚到旧版本，数据库不变（新字段空值处理即可）

// 例如：新增字段
// 1. 先在数据库加字段（可空，有默认值）
ALTER TABLE users ADD COLUMN vip_level INT DEFAULT 0;

// 2. 再部署新版本应用（兼容：有/无字段均可工作）

// 3. 回滚：只需回滚应用（新字段数据不会丢失）

// 可逆迁移（INSERT/DELETE/数据迁移）
// 建议：分步骤执行，每步可独立回滚
async function migrate() {
    // Step 1: 添加新表
    await db.query('CREATE TABLE users_backup LIKE users');
    await db.query('INSERT INTO users_backup SELECT * FROM users');

    // Step 2: 迁移数据到新表（带校验）
    const migrated = await migrateUsers();
    if (migrated.count < original.count) {
        throw new Error('Migration count mismatch, rolling back');
    }

    // Step 3: 原子切换
    await db.query('RENAME TABLE users TO users_old, users_backup TO users');

    // Step 4: 验证无误后删除旧表
    await db.query('DROP TABLE users_old');
}
```

**灰度回滚 vs 全量回滚：**

```
故障发生
    ↓
立即停止金丝雀流量（新版本立即下线）
    ↓
保留旧版本Pod → 金丝雀Pod下线 → 旧版本恢复服务
（金丝雀回滚，30秒内完成）
    ↓
同时排查问题
    ↓
问题修复后 → 重新走 CI/CD 流程发布新版本
```

**回滚决策树：**

```
上线后发现故障
    ↓
故障影响范围？
    ↓
┌─────────────────────┬──────────────────┐
│  < 5% 用户受影响     │  > 5% 用户受影响  │
│  （金丝雀阶段发现）   │  （全量后发现）   │
         ↓                     ↓
   停止金丝雀流量       立即触发全量回滚
   (30秒内完成)         (1-3分钟)
         ↓                     ↓
   观察是否恢复         通知相关团队
         ↓                     ↓
   重新定位问题         复盘 & 修复后重新上线
```

**健康检查 + 自动回滚：**

```yaml

              job="{{args.service-name}}"[1m]))
```

**满分答案：**

**紧急回滚标准流程（5分钟内）：**

```

T+1: 确认故障范围和影响
```

| 恢复验证 | < 1分钟 | 自动化测试 + 监控确认 |
| **总计** | **< 6分钟** | **全链路自动化** |

### 7.10.1 Monorepo 核心概念与 Multirepo 对比

#### 知识点详解

**什么是 Monorepo：**

Monorepo（单一仓库）是一种将多个项目/包放在同一个 Git 仓库中管理的代码组织策略。与之对应的是 Multirepo（多仓库），每个项目独立一个仓库。

```
Multirepo（多仓库）：
  repo-web/          ← 前端应用
  repo-mobile/       ← 移动端应用
  repo-shared/       ← 公共组件库
  repo-utils/        ← 工具函数库

Monorepo（单一仓库）：
  my-monorepo/
  ├── apps/
  │   ├── web/       ← 前端应用
  │   └── mobile/    ← 移动端应用
  ├── packages/
  │   ├── shared/    ← 公共组件库
  │   └── utils/     ← 工具函数库
  └── package.json   ← 根配置
```

**Monorepo vs Multirepo 完整对比：**

| 维度 | Monorepo | Multirepo |
|------|---------|-----------|
| **代码复用** | ✅ 直接引用，无需发布 | ❌ 需发布 npm 包，再安装 |
| **跨包修改** | ✅ 原子提交，一次 PR 改多包 | ❌ 多仓库多 PR，难以同步 |
| **依赖管理** | ✅ 根目录统一管理，版本一致 | ❌ 各仓库独立，版本碎片化 |
| **工程配置** | ✅ 统一 ESLint/TS/CI 配置 | ❌ 各仓库重复配置 |
| **代码搜索** | ✅ 全局搜索，一个 IDE 窗口 | ❌ 多仓库切换，搜索困难 |
| **仓库体积** | ❌ 随项目增多体积增大 | ✅ 各仓库独立，体积小 |
| **权限控制** | ❌ 难以精细控制各包权限 | ✅ 仓库级别权限隔离 |
| **CI 速度** | ❌ 全量构建慢（需增量优化） | ✅ 各仓库独立 CI，互不影响 |
| **学习成本** | ❌ 需要额外工具（Turborepo/Nx） | ✅ 简单直接 |
| **代表项目** | React、Vue、Babel、Vite | 大多数独立项目 |

**Monorepo 适用场景：**

```
✅ 适合 Monorepo：
- 多个项目共享大量公共代码（组件库、工具函数）
- 需要频繁跨项目联动修改（API 变更同时影响前后端）
- 团队规模中等，统一技术栈
- 组件库 + 文档站 + 示例项目

❌ 不适合 Monorepo：
- 项目间完全独立，无代码共享
- 不同团队技术栈差异大
- 仓库体积已经非常大（>10GB）
- 需要精细的权限控制
```

#### 真实面试题

**题目：什么是 Monorepo？它和 Multirepo 有什么区别？各自的优缺点是什么？**

**满分答案：**

**Monorepo** 是将多个相关项目放在同一个 Git 仓库中管理的策略。

**核心优势（3 点）：**
1. **代码复用**：包之间直接引用，无需发布 npm 包，修改立即生效
2. **原子提交**：跨包修改在一个 PR 中完成，保证一致性
3. **统一工程化**：ESLint、TypeScript、CI/CD 配置只需维护一份

**核心劣势（3 点）：**
1. **仓库体积大**：随项目增多，clone 和 CI 时间增加
2. **工具复杂**：需要 Turborepo/Nx 等额外工具管理构建缓存
3. **权限难控**：无法像多仓库那样精细控制各包的访问权限

**选型建议：**
- 共享代码多、跨包联动频繁 → Monorepo
- 项目独立、团队分散 → Multirepo

**Turborepo 缓存原理：**

```
构建时：
  1. 计算输入 hash（源码 + 依赖 + 环境变量）
  2. 检查本地缓存（node_modules/.cache/turbo）
  3. 命中缓存 → 直接还原产物，跳过构建
  4. 未命中 → 执行构建，保存缓存

远程缓存（Remote Cache）：
  - 团队成员共享缓存（CI 和本地共用）
  - 配置：turbo login && turbo link
  - 效果：CI 首次构建后，后续 PR 直接命中缓存
```

### 7.10.3 幽灵依赖（Phantom Dependencies）

#### 知识点详解

**什么是幽灵依赖：**

```
幽灵依赖（Phantom Dependencies）：
  某个包没有在 package.json 中声明，
  但由于 npm/yarn 的扁平化提升（hoisting），
  它被提升到了 node_modules 根目录，
  导致代码可以 require/import 它，
  但这是不安全的（版本可能随时变化）。
```

**幽灵依赖产生原因（npm/yarn 扁平化）：**

```
项目依赖：
  my-app
  └── package-a@1.0.0
      └── lodash@4.17.21  ← 间接依赖

npm/yarn 扁平化后的 node_modules：
  node_modules/
  ├── package-a/
  └── lodash/             ← 被提升到根目录！

结果：
  // my-app 的代码可以直接 require('lodash')
  // 但 lodash 没有在 my-app 的 package.json 中声明
  // 这就是幽灵依赖！
```

**幽灵依赖的危害：**

```javascript
// 危险代码示例
import _ from 'lodash';  // ⚠️ 幽灵依赖！

// 危害：
// 1. package-a 升级后不再依赖 lodash → 你的代码报错
// 2. package-a 升级 lodash 版本 → 你的代码行为改变
// 3. 换用 pnpm 后 → 直接报错（pnpm 不允许访问未声明的依赖）
```

**pnpm 如何解决幽灵依赖：**

```
pnpm 的 node_modules 结构：

node_modules/
├── .pnpm/                    ← 所有依赖的真实存储（硬链接）
│   ├── lodash@4.17.21/
│   │   └── node_modules/
│   │       └── lodash/       ← 真实文件（硬链接到全局 store）
│   └── package-a@1.0.0/
│       └── node_modules/
│           ├── package-a/
│           └── lodash/       ← 符号链接 → .pnpm/lodash@4.17.21
└── package-a/                ← 符号链接 → .pnpm/package-a@1.0.0
                              ← ⚠️ 没有 lodash！（未声明则不可访问）

结果：
  require('lodash')  → ❌ 报错（未在 package.json 声明）
  require('package-a') → ✅ 正常（已声明）
```

**pnpm 依赖提升配置（.npmrc）：**

```ini
# .npmrc（根目录）

# 默认：严格模式，禁止访问未声明的依赖
# node-linker=isolated

# 如果某些工具不兼容 pnpm 的严格模式，可以开启 hoisting
hoist-pattern[]=*eslint*
hoist-pattern[]=*prettier*
hoist-pattern[]=*babel*

# 或者完全开启 hoisting（不推荐，会引入幽灵依赖）
# shamefully-hoist=true
```

#### 真实面试题

**题目：什么是幽灵依赖？pnpm 是如何解决这个问题的？**

**满分答案：**

**幽灵依赖定义：**
项目代码使用了某个包，但该包没有在 `package.json` 中声明。由于 npm/yarn 的扁平化提升，间接依赖被提升到 `node_modules` 根目录，代码可以访问它，但这是不安全的。

**危害：**
- 上游包升级后不再依赖该包 → 代码报错
- 上游包升级该包版本 → 行为不可预期

**pnpm 解决方案：**
pnpm 使用**硬链接 + 符号链接**的存储结构：
- 所有依赖存储在 `.pnpm/` 目录（硬链接到全局 store）
- `node_modules` 根目录**只有**在 `package.json` 中声明的包
- 未声明的间接依赖无法被访问 → 彻底消除幽灵依赖

```bash
# 验证：在 pnpm 项目中访问未声明的依赖
require('lodash')  # ❌ Error: Cannot find module 'lodash'
# 解决：在 package.json 中显式声明
pnpm add lodash
```

### 7.10.4 Monorepo 版本管理与发布（Changesets）

#### 知识点详解

**Changesets 工作流：**

```
开发者提交代码时：
  1. pnpm changeset          ← 描述本次变更（patch/minor/major）
  2. git commit              ← 提交代码 + changeset 文件

发布时（CI 或手动）：
  3. pnpm changeset version  ← 根据 changeset 更新版本号
  4. pnpm changeset publish  ← 发布到 npm
```

**安装与初始化：**

```bash

```

```json
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

**创建 Changeset（开发者操作）：**

```bash

# 交互式选择：
# 1. 选择哪些包有变更
# 2. 选择变更类型（patch/minor/major）
# 3. 填写变更描述

# 生成文件：.changeset/random-name.md

# ---

```

**版本更新与发布：**

```bash

pnpm changeset version
# 效果：
# - 更新各包的 package.json version
# - 生成/更新 CHANGELOG.md
# - 删除已处理的 changeset 文件

# 发布到 npm
pnpm changeset publish
# 效果：
# - 执行 pnpm publish（各包）
# - 创建 git tag
```

**GitHub Actions 自动发布：**

```yaml

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v2
        with:
          version: 8

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
          registry-url: 'https://registry.npmjs.org'

      - run: pnpm install --frozen-lockfile

      - name: Create Release Pull Request or Publish
        uses: changesets/action@v1
        with:
          publish: pnpm changeset publish
          version: pnpm changeset version
          commit: "chore(release): version packages"
          title: "chore(release): version packages"
```

**Changesets 工作流说明：**

```
PR 合并到 main 时：
  - 如果有 changeset 文件 → 自动创建 "Version Packages" PR
  - 合并 "Version Packages" PR → 自动发布到 npm

效果：
  - 开发者只需关注写 changeset
  - 版本号更新和发布完全自动化
```

**版本号规范（Semantic Versioning）：**

```
版本格式：MAJOR.MINOR.PATCH（如 1.2.3）

patch（补丁）：1.0.0 → 1.0.1
  - Bug 修复，向后兼容

minor（次版本）：1.0.0 → 1.1.0
  - 新增功能，向后兼容

major（主版本）：1.0.0 → 2.0.0
  - 破坏性变更（Breaking Change）

预发布版本：
  1.0.0-alpha.1  ← 内部测试
  1.0.0-beta.1   ← 公测
  1.0.0-rc.1     ← 候选发布
```

#### 真实面试题

**题目：Monorepo 中如何管理多包的版本发布？Changesets 的工作流是什么？**

**满分答案：**

**Changesets 工作流（4 步）：**

```
1. 开发者提交代码时运行 pnpm changeset
   → 选择变更的包 + 变更类型（patch/minor/major）+ 描述

2. 生成 .changeset/xxx.md 文件，随代码一起提交

3. CI 检测到 changeset 文件，自动创建 "Version Packages" PR
   → 更新各包 package.json 版本号 + 生成 CHANGELOG.md

4. 合并 PR → 自动 pnpm publish 发布到 npm
```

**为什么用 Changesets 而不是手动管理版本：**
- 自动追踪哪些包有变更
- 自动计算版本号（基于 semver）
- 自动生成 CHANGELOG
- 与 GitHub Actions 深度集成，发布完全自动化

#### 真实面试题

**题目：在 Monorepo 中，如何处理包之间的循环依赖？如何保证构建顺序正确？**

**满分答案：**

**循环依赖处理：**

```
问题：
  packages/a → 依赖 packages/b
  packages/b → 依赖 packages/a  ← 循环！

解决方案：
  1. 提取公共逻辑到 packages/shared
  2. a 和 b 都依赖 shared，不互相依赖

  packages/a → packages/shared
  packages/b → packages/shared
```

**构建顺序保证（Turborepo）：**

```json
// turbo.json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"]  // ^ 表示先构建所有依赖包
    }
  }
}
```

```
执行 turbo run build 时：
  1. 分析依赖图
  2. 先构建 packages/utils（无依赖）
  3. 再构建 packages/ui（依赖 utils）
  4. 最后构建 apps/web（依赖 ui + utils）
  5. 同级包并行构建（充分利用 CPU）
```

**本地开发时的联动（watch 模式）：**

```bash

# 或者只启动 web 应用及其依赖的 watch
pnpm --filter @myapp/web... run dev
```

```javascript
class PromptManager {
    constructor() {
        this.prompts = new Map();
        this.versions = new Map();
    }
    
    // 注册Prompt
    register(name, versions) {
        this.prompts.set(name, versions);
    }
    
    // 获取Prompt（支持版本切换和A/B测试）
    get(name, options = {}) {
        const versions = this.prompts.get(name);
        if (!versions) throw new Error(`Prompt ${name} not found`);
        
        // A/B测试
        if (options.experiment) {
            return this.getExperimentVersion(name, options.experiment);
        }
        
        // 指定版本
        if (options.version) {
            return versions.find(v => v.version === options.version);
        }
        
        // 默认最新版本
        return versions[versions.length - 1];
    }
    
    // 渲染Prompt（变量替换）
    render(prompt, variables) {
        return prompt.content.replace(/\{\{(\w+)\}\}/g, (match, key) => {
            return variables[key] ?? match;
        });
    }
}
```

```javascript
class PromptABTest {
    constructor() {
        this.experiments = new Map();
    }
    
    // 创建实验
    createExperiment(name, variants) {
        this.experiments.set(name, {
            variants,
            results: new Map()
        });
    }
    
    // 获取实验版本（根据用户ID分流）
    getVariant(experimentName, userId) {
        const experiment = this.experiments.get(experimentName);
        if (!experiment) return null;
        
        // 哈希分流
        const hash = this.hash(userId + experimentName);
        const index = hash % experiment.variants.length;
        return experiment.variants[index];
    }
    
    // 记录实验结果
    track(experimentName, variant, metric) {
        const experiment = this.experiments.get(experimentName);
        if (!experiment.results.has(variant)) {
            experiment.results.set(variant, []);
        }
        experiment.results.get(variant).push(metric);
    }
    
    // 获取实验报告
    getReport(experimentName) {
        const experiment = this.experiments.get(experimentName);
        const report = {};
        
        experiment.variants.forEach(variant => {
            const metrics = experiment.results.get(variant) || [];
            report[variant] = {
                count: metrics.length,
                avgScore: metrics.reduce((a, b) => a + b, 0) / metrics.length
            };
        });
        
        return report;
    }
    
    hash(str) {
        let hash = 0;
        for (let i = 0; i < str.length; i++) {
            const char = str.charCodeAt(i);
            hash = ((hash << 5) - hash) + char;
            hash = hash & hash;
        }
        return Math.abs(hash);
    }
}
```

```javascript
class TokenMonitor {
    constructor() {
        this.metrics = [];
    }
    
    // 记录一次API调用
    log(request) {
        const metric = {
            timestamp: Date.now(),
            model: request.model,
            inputTokens: request.inputTokens,
            outputTokens: request.outputTokens,
            cost: this.calculateCost(request.model, request.inputTokens, request.outputTokens),
            promptVersion: request.promptVersion,
            userId: request.userId
        };
        
        this.metrics.push(metric);
        
        // 上报到监控系统
        this.report(metric);
    }
    
    calculateCost(model, inputTokens, outputTokens) {
        const pricing = {
            'gpt-4': { input: 0.03, output: 0.06 },      // $/1K tokens
            'gpt-3.5-turbo': { input: 0.0015, output: 0.002 }
        };
        
        const price = pricing[model];
        if (!price) return 0;
        
        return (inputTokens / 1000 * price.input) + 
               (outputTokens / 1000 * price.output);
    }
    
    // 获取统计报告
    getReport(timeRange = '1d') {
        const now = Date.now();
        const range = { '1h': 3600000, '1d': 86400000, '7d': 604800000 }[timeRange];
        
        const recent = this.metrics.filter(m => now - m.timestamp < range);
        
        return {
            totalCalls: recent.length,
            totalTokens: recent.reduce((sum, m) => sum + m.inputTokens + m.outputTokens, 0),
            totalCost: recent.reduce((sum, m) => sum + m.cost, 0),
            avgCostPerCall: recent.reduce((sum, m) => sum + m.cost, 0) / recent.length
        };
    }
}
```
