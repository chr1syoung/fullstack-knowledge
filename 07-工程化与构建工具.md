## 7.1 Webpack

### 7.1.1 Vue CLI 核心功能

**Vue CLI 是 Vue.js 官方提供的脚手架工具：**

1. **项目脚手架**
   - 创建 Vue 项目基础结构
   - 选择配置（TypeScript、CSS 预处理器等）

2. **开发服务器**
   - 内置 webpack-dev-server
   - 热模块替换（HMR）
   - 自动打开浏览器

3. **构建打包**
   - 代码打包、压缩、优化
   - Code Splitting（代码分割）
   - Tree Shaking（清除未使用代码）

4. **加载器配置**
   - babel-loader（JS 语法转换）
   - css-loader / sass-loader
   - file-loader / url-loader

5. **插件系统**
   - 内置插件：HtmlWebpackPlugin、MiniCssExtractPlugin
   - 支持自定义插件

6. **图形化配置**
   - `vue ui` 命令打开图形界面
   - 可视化配置 webpack

**Vue CLI vs Vite：**

| 特性 | Vue CLI | Vite |
|------|---------|------|
| 构建速度 | 较慢（全量构建） | 快（按需加载） |
| 开发服务器 | webpack-dev-server | 原生 ESM |
| HMR | 较慢 | 极快 |
| 生产构建 | webpack | Rollup |

**题目：Vue CLI 提供了哪些功能？它和 Vite 有什么区别？**

**满分答案：**

**Vue CLI 功能：**
1. 项目脚手架生成
2. 开发服务器和热更新
3. 生产环境构建打包
4. 插件和加载器配置
5. 图形化配置界面
6. 单元测试和 E2E 测试支持

**Vue CLI vs Vite：**
- Vite 是新一代构建工具，更快
- Vue CLI 基于 webpack，更成熟
- 新项目推荐 Vite，老项目继续用 Vue CLI

### 7.1.2 核心概念

#### 知识点详解

Webpack 是一个现代 JavaScript 应用程序的静态模块打包工具。

**核心概念：**

1. **Entry（入口）**：Webpack 开始构建的起点
2. **Output（输出）**：打包后文件的输出位置
3. **Loader**：处理非 JavaScript 文件（CSS、图片等）
4. **Plugin**：扩展 Webpack 功能
5. **Mode**：开发/生产模式

**基础配置：**

```javascript
// webpack.config.js
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');

module.exports = {
    // 入口
    entry: {
        main: './src/index.js',
        vendor: './src/vendor.js'
    },
    
    // 输出
    output: {
        path: path.resolve(__dirname, 'dist'),
        filename: '[name].[contenthash].js',
        chunkFilename: '[name].[contenthash].chunk.js',
        clean: true  // 每次构建前清理 dist
    },
    
    // 模式
    mode: 'production',  // 'development' | 'production' | 'none'
    
    // Source Map
    devtool: 'source-map',
    
    // 模块解析
    resolve: {
        extensions: ['.js', '.jsx', '.ts', '.tsx'],
        alias: {
            '@': path.resolve(__dirname, 'src')
        }
    },
    
    // Loader
    module: {
        rules: [
            // JavaScript/TypeScript
            {
                test: /\.(js|jsx|ts|tsx)$/,
                exclude: /node_modules/,
                use: {
                    loader: 'babel-loader',
                    options: {
                        presets: [
                            '@babel/preset-env',
                            '@babel/preset-react',
                            '@babel/preset-typescript'
                        ]
                    }
                }
            },
            
            // CSS
            {
                test: /\.css$/,
                use: [
                    MiniCssExtractPlugin.loader,  // 生产环境提取 CSS
                    // 'style-loader',            // 开发环境注入 style 标签
                    'css-loader',
                    'postcss-loader'
                ]
            },
            
            // SCSS
            {
                test: /\.scss$/,
                use: [
                    MiniCssExtractPlugin.loader,
                    'css-loader',
                    'postcss-loader',
                    'sass-loader'
                ]
            },
            
            // 图片
            {
                test: /\.(png|jpg|gif|svg|webp)$/,
                type: 'asset',
                parser: {
                    dataUrlCondition: {
                        maxSize: 8 * 1024  // 8KB 以下转 base64
                    }
                },
                generator: {
                    filename: 'images/[name].[hash][ext]'
                }
            },
            
            // 字体
            {
                test: /\.(woff|woff2|eot|ttf|otf)$/,
                type: 'asset/resource',
                generator: {
                    filename: 'fonts/[name].[hash][ext]'
                }
            }
        ]
    },
    
    // Plugin
    plugins: [
        new HtmlWebpackPlugin({
            template: './public/index.html',
            filename: 'index.html',
            minify: {
                removeComments: true,
                collapseWhitespace: true
            }
        }),
        new MiniCssExtractPlugin({
            filename: 'css/[name].[contenthash].css'
        })
    ],
    
    // 优化
    optimization: {
        splitChunks: {
            chunks: 'all',
            cacheGroups: {
                vendor: {
                    test: /[\\/]node_modules[\\/]/,
                    name: 'vendors',
                    chunks: 'all'
                }
            }
        },
        runtimeChunk: 'single'
    },
    
    // 开发服务器
    devServer: {
        port: 3000,
        hot: true,
        historyApiFallback: true,
        proxy: {
            '/api': {
                target: 'http://localhost:8080',
                changeOrigin: true
            }
        }
    }
};
```

```javascript
// my-loader.js
module.exports = function(source) {
    // source 是文件内容
    const result = source.replace(/console\.log\(.*?\);?/g, '');

// 异步 Loader
module.exports = function(source) {
    const callback = this.async();
    
    someAsyncOperation(source, (err, result) => {
        if (err) return callback(err);
        callback(null, result);
    });
```

### 7.1.3 自定义 Plugin

```javascript
// MyPlugin.js
class MyPlugin {
    constructor(options) {
        this.options = options;
    }
    
    apply(compiler) {
        // 在 emit 钩子（输出文件前）执行
        compiler.hooks.emit.tapAsync('MyPlugin', (compilation, callback) => {
            // 获取所有输出文件
            const assets = compilation.assets;
            
            // 创建一个文件清单
            let manifest = '';
            for (const filename in assets) {
                manifest += `${filename}\n`;
            }
            
            // 添加到输出
            compilation.assets['manifest.txt'] = {
                source: () => manifest,
                size: () => manifest.length
            };
            
            callback();
        });
    }
}

module.exports = MyPlugin;
```

## 7.2 Vite

### 7.2.1 Vite 原理

**为什么 Vite 快：**

1. **开发环境**：
   - 利用浏览器原生 ES Modules
   - 按需编译，不打包
   - 使用 esbuild 预构建依赖（比 Babel 快 10-100 倍）

2. **生产环境**：
   - 使用 Rollup 打包
   - 更好的 Tree Shaking

```javascript
// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import vue from '@vitejs/plugin-vue';
import path from 'path';

export default defineConfig({
    plugins: [
        react(),
        // vue()
    ],
    
    resolve: {
        alias: {
            '@': path.resolve(__dirname, './src')
        }
    },
    
    server: {
        port: 3000,
        proxy: {
            '/api': {
                target: 'http://localhost:8080',
                changeOrigin: true,
                rewrite: (path) => path.replace(/^\/api/, '')
            }
        }
    },
    
    build: {
        outDir: 'dist',
        sourcemap: true,
        rollupOptions: {
            output: {
                manualChunks: {
                    vendor: ['react', 'react-dom'],
                    router: ['react-router-dom']
                }
            }
        }
    },
    
    css: {
        preprocessorOptions: {
            scss: {
                additionalData: `@import "@/styles/variables.scss";`
            }
        }
    }
});
```

```javascript
// my-vite-plugin.js
export default function myPlugin() {

            if (id.endsWith('.js')) {
                return code.replace(/console\.log\(.*?\);?/g, '');

        // 构建结束

            console.log('Build ended');
        }
    };
}
```

### 7.2.2 Vite 深入：为什么比 Webpack 快

#### 知识点详解

**Webpack 构建流程：**

```
1. 读取配置文件
2. 递归分析依赖（AST）
3. 打包成一个/多个 bundle
4. 代码转换（Babel/TypeScript）
5. 代码压缩/优化
6. 输出到 dist

问题：每次修改都要重新打包整个项目
```

**Vite 构建原理：**

```
开发环境：
1. 启动开发服务器
2. 不打包！直接用原生 ESM
3. 浏览器请求哪个模块，Vite 就编译哪个
4. 依赖用 esbuild 预构建（.js → .esm.js）

生产环境：
1. 用 Rollup 打包（比 Webpack 快）
2. Tree Shaking 更彻底
3. 代码分割更智能
```

**Vite 快的核心原因：**

```javascript
// 1. ESM 按需加载（无需打包）
// 浏览器原生支持
import { add } from './math.js';  // 直接请求这个文件

// Vite 拦截请求，现场编译
// 返回：export function add() {}

// 2. esbuild 预构建（极快）
// 第一次启动时，把 node_modules 里的 CommonJS 转成 ESM
// esbuild 是 Go 写的，比 Babel 快 10-100 倍

// 3. 依赖预构建
// import { Button } from 'antd'
// ↓ 预构建后
// import Button from '/node_modules/.vite/antd/esm/Button.js'

// 4. 热更新快
// Webpack: 修改 → 重新打包 → 页面刷新
// Vite: 修改 → 只更新那个模块 → HRM 局部更新
```

**Vite vs Webpack 对比：**

| 特性 | Webpack | Vite |
|------|---------|------|
| 开发启动 | 慢（需打包） | 快（不打包） |
| 热更新 | 较慢 | 极快 |
| 构建速度 | 慢 | 快（Rollup） |
| 依赖预构建 | 无 | esbuild |
| ESM 支持 | 需要转换 | 原生支持 |
| 生态 | 成熟丰富 | 较新 |

**适用场景：**

```javascript
// 适合 Vite
- Vue 3 项目
- React 项目（用 @vitejs/plugin-react）
- 快速原型开发
- 小型项目

// 仍需 Webpack
- 复杂的大型项目
- 需要大量自定义配置
- 老项目迁移
- 特殊加载器需求
```

#### 真实面试题

**题目：Vite 为什么比 Webpack 快？它的构建原理是什么？**

**满分答案：**

**开发环境快的原因：**
1. **不打包**：利用浏览器原生 ESM，按需编译
2. **esbuild 预构建**：Go 语言写的编译器，比 Babel 快 10-100 倍
3. **HMR 局部更新**：只更新变化的模块，不重整个页面

**生产环境快的原因：**
1. **Rollup 打包**：比 Webpack 更适合库打包
2. **更好的 Tree Shaking**：基于 ESM 分析

**核心区别：**
- Webpack：先打包再运行
- Vite：先运行，按需编译

### 7.3.1 ESLint

**ESLint 检查过程：**

```
1. 解析 → 2. 规则匹配 → 3. 报告错误 → 4. 修复/阻止提交
```

**1. 解析阶段：**

```javascript
// ESLint 使用 Espree 解析代码
const espree = require('espree');
const ast = espree.parse(code, {
    ecmaVersion: 2020,
    sourceType: 'module'
});
```

**2. 规则匹配：**

```javascript
// 规则结构
const rule = {
    meta: {
        type: 'problem',
        docs: { description: 'disallow var' }
    },
    create(context) {
        return {
            VariableDeclaration(node) {
                if (node.kind === 'var') {
                    context.report({
                        node,
                        message: 'Use let or const instead of var'
                    });
                }
            }
        };
    }
};
```

#### 真实面试题

**题目：ESLint 的检查过程是什么？它是如何发现代码问题的？**

**满分答案：**

**ESLint 检查过程：**

1. **解析代码**：将 JS 代码解析为 AST（抽象语法树）
2. **加载配置**：读取 .eslintrc.js 等配置文件
3. **加载规则**：根据配置加载对应规则
4. **遍历 AST**：使用 visitor 模式遍历语法树
5. **规则匹配**：每个规则检查对应节点
6. **报告错误**：发现问题使用 context.report 报告
7. **自动修复**：使用 --fix 自动修复可修复的问题

**集成方式：**
- VS Code 保存时自动检查
- Git Hook 提交前检查
- CI/CD 流水线检查

### 7.6.2 Git Flow

```
main          ─────────────────────────────────────────────────────
                    ↑                                    ↑
release       ──────┤                                    │
                    │                                    │
develop       ──────┴────────────────────────────────────┤
                    ↑              ↑                     │

                                   │                     │
```

**分支说明：**

- `hotfix/*`：热修复分支，从 main 创建，修复后合并到 main 和 develop

#### 真实面试题

**题目：Webpack 和 Vite 有什么区别？为什么 Vite 开发环境更快？**

**满分答案：**

**Webpack 的工作方式：**

1. 从入口文件开始，递归分析所有依赖
2. 将所有模块打包成一个或多个 bundle
3. 开发服务器启动时需要完整打包
4. 每次修改都需要重新打包（虽然有 HMR 优化）

**Vite 的工作方式：**

1. **开发环境**：
   - 利用浏览器原生 ES Modules
   - 不打包，按需编译
   - 使用 esbuild 预构建 node_modules（比 Babel 快 10-100 倍）
   - 修改文件时只需重新编译该文件

2. **生产环境**：
   - 使用 Rollup 打包
   - 更好的 Tree Shaking

**性能对比：**

| 指标 | Webpack | Vite |
|------|---------|------|
| 冷启动 | 慢（需要打包） | 快（按需编译） |
| HMR | 较慢 | 极快 |
| 构建速度 | 较慢 | 快（esbuild） |
| 生产构建 | 成熟稳定 | 较新 |
| 生态 | 非常丰富 | 快速增长 |

**为什么 Vite 开发环境快：**

```
Webpack 开发流程：
源码 → 打包 → bundle.js → 浏览器

Vite 开发流程：
源码 → 浏览器请求 → 按需编译 → 返回 ES Module
```

1. **无需打包**：浏览器直接加载 ES Modules
2. **esbuild 预构建**：Go 语言编写，比 JS 快 10-100 倍
3. **按需编译**：只编译当前页面需要的文件
4. **精准 HMR**：只更新修改的模块，不影响其他模块

**选择建议：**

- 新项目：优先选择 Vite
- 已有 Webpack 项目：评估迁移成本
- 需要复杂自定义配置：Webpack 更成熟
- 追求开发体验：Vite 更好

### 7.x.x package.json 中 sideEffects 属性的作用 [必会]

#### 知识点详解

`sideEffects` 是 Webpack 4+ 新增的配置项，用于告诉 Webpack 哪些文件是"纯函数"（无副作用），从而支持 **Tree Shaking** 优化。

**基本用法：**

```json
// package.json
{
  "sideEffects": [
    "*.css",
    "*.less",
    "./src/some-module.js"
  ]
}
```

**两种配置方式：**

| 配置 | 含义 |
|------|------|
| `sideEffects: false` | 所有文件都是纯函数，可以安全地 Tree Shaking |
| `sideEffects: [...]` | 明确标记哪些文件有副作用（保留），其他文件可摇树 |

**实际例子：**

```js
// math.js — 纯函数，无副作用
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }

// ui.js — 有副作用（注册全局组件）
export const Button = () => {...};
window.MyUI = { Button }; // ← 副作用！这行代码不能删除

// index.js
export { add, multiply } from './math';  // 可被 Tree Shaking
export { Button } from './ui';            // ui.js 整体被保留（因为有副作用）
```

```json
{
  "sideEffects": ["./src/ui.js", "*.css"]
}
```

**为什么 CSS 也需要标记？**

```js
// 如果不标记 *.css 为有副作用，Webpack 会认为可以安全删除
import './style.css';
// 如果没有 "*.css" 在 sideEffects 中，这行导入会被 Tree Shaking 删掉！
```

**Tree Shaking 的工作原理：**

```
源代码          标记后           Tree Shaking 后
add(a,b)     →  add(a,b)       →  add(a,b)     (保留)
multiply()   →  multiply()      →  (删除)       (未使用)
Button       →  Button (有副作用) → Button       (保留)
```

**常见面试题：**

> **Q：为什么 CSS 文件要加入 sideEffects？**
> 因为 CSS 文件本身不导出任何内容，只通过 `@import` 或 `import` 被引入。如果不标记，Webpack 会认为它"没用"而删除掉。

> **Q：sideEffects 和 externals 有什么区别？**
> - `sideEffects`：优化产物大小，告诉 Webpack 哪些文件可以安全删除
> - `externals`：告诉 Webpack 不打包某些库（从外部 CDN/Node_modules 加载）

#### 面试加分项
- 了解 Tree Shaking 的前提：必须是 ES Module（静态 import/export），CommonJS 不支持
- 知道在 package.json 设置 `"sideEffects": false` 意味着所有模块都是纯函数

## 7.7 Git pull 和 Git fetch 区别

#### 知识点详解

**Git Fetch：**

```bash
# 获取远程更新，但不合并
git fetch origin

# 获取所有分支
git fetch --all

# 获取并查看差异
git fetch -v
```

**Git Pull：**

```bash
# 获取并合并远程更新
git pull origin main

# 相当于 git fetch + git merge
git pull = git fetch + git merge

# 使用 rebase
git pull --rebase origin main
```

**核心区别图解：**

```
git fetch:
┌─────────────┐     ┌─────────────┐
│   Remote    │ ──→ │   Local     │
│  origin/main│     │  origin/main│  (更新了，但本地 main 没变)
└─────────────┘     └─────────────┘

git pull:
┌─────────────┐     ┌─────────────┐
│   Remote    │ ──→ │   Local     │
│  origin/main│     │  origin/main│ ──→ main (自动合并)
└─────────────┘     └─────────────┘
```

**最佳实践：**

```bash
# 1. 先看差异
git fetch origin
git log HEAD..origin/main --oneline

# 2. 确认安全后再合并
git merge origin/main

# 3. 或者使用 rebase 保持线性历史
git pull --rebase origin main
```

#### 真实面试题

**题目：git pull 和 git fetch 有什么区别？**

**满分答案：**

| 操作 | git fetch | git pull |
|------|----------|---------|
| 获取远程更新 | ✅ | ✅ |
| 自动合并 | ❌ | ✅ |
| 本地分支更新 | ❌ | ✅ |
| 安全性 | 高（可先查看） | 中（直接合并） |

**推荐工作流：**
1. `git fetch` 获取远程更新
2. `git diff` 查看差异
3. 确认无误后 `git merge` 或 `git pull`
