## 12.3 JSBridge 实现原理

#### 知识点详解

**什么是 JSBridge？**

JSBridge 是 H5 与 Native（iOS/Android）之间通信的桥梁，让 JS 能调用原生能力（相机、定位、支付等），Native 也能调用 JS 方法。

```
H5（WebView）  ←→  JSBridge  ←→  Native App
```

**JS 调用 Native 的两种方式：**

**方式一：URL Scheme 拦截**

```javascript
// H5 端：构造特殊 URL
function callNative(method, params, callback) {
    const callbackId = `cb_${Date.now()}`;

    // 注册回调
    window._jsbridge_callbacks = window._jsbridge_callbacks || {};
    window._jsbridge_callbacks[callbackId] = callback;

    // 构造 URL Scheme
    const url = `myapp://jsbridge?method=${method}&params=${encodeURIComponent(JSON.stringify(params))}&callbackId=${callbackId}`;

    // 触发 URL（Native 拦截）
    const iframe = document.createElement('iframe');
    iframe.style.display = 'none';
    iframe.src = url;
    document.body.appendChild(iframe);
    setTimeout(() => document.body.removeChild(iframe), 100);
}

// Native 回调 JS
function _nativeCallback(callbackId, result) {
    const cb = window._jsbridge_callbacks[callbackId];
    if (cb) {
        cb(result);
        delete window._jsbridge_callbacks[callbackId];
    }
}

// 使用
callNative('getLocation', {}, (result) => {
    console.log('位置:', result.latitude, result.longitude);
});
```

**方式二：注入 JS 对象（更现代）**

```javascript
// Android：WebView.addJavascriptInterface()
// iOS：WKScriptMessageHandler

// Native 注入后，H5 可以直接调用
window.NativeBridge.getLocation(JSON.stringify({
    success: 'window._locationCallback'
}));

// 或者使用 iOS 的 webkit.messageHandlers
window.webkit.messageHandlers.getLocation.postMessage({
    callbackId: 'cb_001'
});
```

**Native 调用 JS：**

```javascript
// Native 端执行 JS 代码
// Android: webView.evaluateJavascript("javascript:方法名(参数)", null)
// iOS: webView.evaluateJavaScript("方法名(参数)")

// H5 端暴露方法供 Native 调用
window.onNativeMessage = function(data) {
    const { type, payload } = JSON.parse(data);
    switch (type) {
        case 'userInfo':
            updateUserInfo(payload);
            break;
        case 'networkChange':
            handleNetworkChange(payload);
            break;
    }
};
```

**完整 JSBridge SDK 封装：**

```javascript
class JSBridge {
    constructor() {
        this.callbacks = new Map();
        this.callbackId = 0;

        // 监听 Native 消息
        window.addEventListener('message', this._handleNativeMessage.bind(this));
    }

    // JS 调用 Native
    call(method, params = {}) {
        return new Promise((resolve, reject) => {
            const id = `cb_${++this.callbackId}`;
            this.callbacks.set(id, { resolve, reject });

            const message = JSON.stringify({ method, params, callbackId: id });

            if (window.webkit?.messageHandlers?.bridge) {
                // iOS
                window.webkit.messageHandlers.bridge.postMessage(message);
            } else if (window.AndroidBridge) {
                // Android
                window.AndroidBridge.call(message);
            } else {
                // 降级：模拟或报错
                reject(new Error('非 App 环境'));
            }

            // 超时处理
            setTimeout(() => {
                if (this.callbacks.has(id)) {
                    this.callbacks.delete(id);
                    reject(new Error(`${method} 调用超时`));
                }
            }, 10000);
        });
    }

    // 处理 Native 回调
    _handleNativeMessage(event) {
        const { callbackId, result, error } = JSON.parse(event.data);
        const cb = this.callbacks.get(callbackId);
        if (!cb) return;

        this.callbacks.delete(callbackId);
        error ? cb.reject(new Error(error)) : cb.resolve(result);
    }
}

// 使用
const bridge = new JSBridge();

async function getLocation() {
    try {
        const location = await bridge.call('getLocation');
        console.log(location);
    } catch (e) {
        console.error('获取位置失败:', e.message);
    }
}
```

#### 真实面试题

**题目：在 Hybrid 开发中，JSBridge 的实现原理是什么？**

**满分答案：**

**JS 调用 Native 两种方式：**
1. **URL Scheme 拦截**：H5 构造特殊 URL，Native WebView 拦截并处理
2. **注入 JS 对象**：Native 向 WebView 注入对象（Android `addJavascriptInterface`，iOS `WKScriptMessageHandler`）

**Native 调用 JS：**
- 直接执行 JS 代码：`evaluateJavaScript("window.onMessage(data)")`

**完整通信流程：**
1. H5 调用 → 生成 callbackId → 发消息给 Native
2. Native 处理 → 执行 `evaluateJavaScript` 回调 H5
3. H5 根据 callbackId 找到对应 Promise，resolve/reject

## 12.4 React Native 核心详解

### 12.4.1 React Native 基础与核心原理

#### 知识点详解

**React Native 是什么：**

React Native（RN）是 Facebook 于 2015 年开源的跨平台移动应用框架，使用 JavaScript 和 React 语法编写原生应用，同时支持 iOS 和 Android。

**React Native vs React Web 核心区别：**

| 维度 | React Web | React Native |
|------|-----------|-------------|
| 渲染目标 | DOM（HTML） | 原生组件（UIView / Android View） |
| 组件库 | div / span / p | View / Text / Image / TextInput |
| 样式 | CSS | 类似 CSS-in-JS（Flexbox） |
| 布局引擎 | 浏览器渲染 | Yoga（C++ 跨平台布局引擎） |
| 事件系统 | DOM Events | 统一封装为 Touchable 系列 |
| 网络请求 | fetch / axios | 相同（Native 层实现） |
| 路由 | react-router | react-navigation |
| 状态管理 | Redux / Context | 相同（React 生态完全兼容） |

**React Native 核心组件：**


**React Native 运行原理：**

```
┌─────────────────────────────────────────────┐
│              React Native App               │
├─────────────────────────────────────────────┤
│  JavaScript Bundle（JS 代码）               │
│       ↓ 调用          ↑ 结果返回             │
│  JavaScript Engine（Hermes / JSC）          │
│       ↓ JSI 通信                            │
│  Native Modules（C++/Swift/Java）            │
│       ↓                                      │
│  原生 UI（iOS UIView / Android View）       │
└─────────────────────────────────────────────┘
```

#### 真实面试题

**题目：React Native 和 React Web 有什么区别？为什么 RN 能用 React 语法开发原生应用？**

**满分答案：**

**核心区别：**

| 维度 | React Web | React Native |
|------|----------|-------------|
| 渲染引擎 | 浏览器 DOM | 原生组件（iOS UIView / Android View） |
| 布局 | CSS | CSS-subset + Flexbox（Yoga） |
| 组件 | div/span/button | View/Text/TouchableOpacity |
| 调试 | Chrome DevTools | Flipper / Chrome DevTools |

**RN 能开发原生应用的原因：**

1. **Hermes / JSCore**：App 内置 JS 引擎，运行 JS 代码
2. **JSI（JavaScript Interface）**：JS 与原生代码的通信桥梁
3. **React 组件映射**：JSX 中 `<View>` 实际对应原生 UIView/View
4. **Yoga 布局引擎**：C++ 实现，跨 iOS/Android 统一 Flexbox 布局

**JSX 到原生组件的映射过程：**
```
JSX: <View><Text>Hello</Text></View>
  ↓ Babel 转译（JSX → React.createElement）
JS 对象: { type: 'View', props: { children: [...] } }
  ↓ JSI/Bridge 传递给原生层
原生: UIStackView > UILabel(text="Hello")
```

### 12.4.2 React Native 新架构（Fabric / TurboModules / JSI / Hermes）

#### 知识点详解

**新架构三大核心组件：**

```
旧架构（0.68 之前）                 新架构（0.76+ 默认）
┌──────────────────┐              ┌──────────────────────┐
│   JS Thread      │              │   JS Thread            │
│  (React 逻辑)     │              │  (React 逻辑)          │
└───────┬──────────┘              └───────────┬──────────┘
        │                                       │
        │ Bridge（异步 JSON）                    │ JSI（同步直接调用）
        ▼                                       ▼
┌──────────────────┐              ┌──────────────────────┐
│  Native Modules   │              │  TurboModules（懒加载）│
│                  │              │  Fabric（同步渲染）     │
└──────────────────┘              └──────────────────────┘
```

**JSI（JavaScript Interface）：**

```cpp
// JSI 的核心：允许 JS 直接调用 C++/Java/Swift 对象
// 旧架构：JS → JSON序列化 → Bridge → 原生（性能损耗大）
// 新架构：JS → 直接调用（无序列化开销）

// TurboModule 调用示例（同步）
// MyNativeModule.ts
import { TurboModule } from 'react-native';
export interface Spec extends TurboModule {
    getDeviceName(): string;
    multiply(a: number, b: number): number;
}
import MyNativeModule from './MyNativeModule';
const result = MyNativeModule.multiply(3, 4); // 同步调用，无延迟
```

**Fabric 渲染器（新渲染引擎）：**

```jsx
// Fabric 带来的关键能力

// 1. 同步渲染：UI 更新同步执行，无异步延迟
// 旧架构：setState → 异步通知原生 → 异步渲染（有延迟）
// 新架构：setState → 同步调用原生渲染（无延迟）

// 2. 并发渲染（Concurrent React）
import React, { useTransition, Suspense } from 'react';

function SearchResults({ query }) {
    const [isPending, startTransition] = useTransition();

    const handleChange = (text) => {
        startTransition(() => {
            setQuery(text); // 这个更新被标记为可中断，不阻塞 UI
        });
    };

    return (
        <View>
            <TextInput onChangeText={handleChange} />
            {isPending ? <ActivityIndicator /> : <Results query={query} />}
        </View>
    );
}

// 3. 新版 Pressable 支持精确按压状态
<Pressable
    onPressIn={() => console.log('按下了')}
    onPressOut={() => console.log('松开了')}
    onLongPress={() => console.log('长按')}
>
    <Text>按住我</Text>
</Pressable>
```

**Hermes 引擎：**

```javascript
// Hermes：Facebook 自研 JS 引擎，RN 0.70+ 默认

// Hermes 的优势：
// 1. AOT 编译（Ahead-of-Time）：JS → Hermes Bytecode
//    启动时直接执行，无需解析
//    冷启动时间减少 30-50%
//    内存占用降低 20-30%
// 2. 优化了 GC，减少卡顿
// 3. 内置 source-map 支持，调试友好

// metro.config.js 配置 Hermes
module.exports = {
    transformer: {
        getTransformOptions: async () => ({
            transform: {
                experimentalImportSupport: false,
                inlineRequires: true,
            },
        }),
    },
};
```

**TurboModules（模块懒加载）：**

```typescript
// TurboModules vs 旧 Native Modules
// 旧架构：App 启动时全部初始化 Native Modules
// TurboModules：按需加载，只初始化当前需要的模块

// 定义 TurboModule（TypeScript）
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
    getApiKey(): string;
    fetchData(url: string): Promise<string>;
}

export default TurboModuleRegistry.getEnforcing<Spec>('MyNativeModule');
```

**新旧架构对比总结：**

| 维度 | 旧架构 | 新架构 |
|------|--------|--------|
| JS 与原生通信 | Bridge（异步 JSON） | JSI（同步直接调用） |
| 渲染模式 | 异步渲染 | 同步渲染 |
| 线程模型 | JS/UI/Shadow 三线程 | JS 与 UI 线程可直接通信 |
| 模块加载 | 全量初始化 | 懒加载（TurboModules） |
| 列表滚动帧率 | 40-50 FPS | 55-60 FPS |
| 启动速度 | 较慢 | 减少 30-50% |
| 并发渲染 | 不支持 | 支持（React 18+） |

#### 真实面试题

**题目：React Native 新架构（Fabric / TurboModules / Hermes）与旧架构的区别是什么？新架构解决了哪些问题？**

**满分答案：**

**旧架构的问题：**

1. **Bridge 性能瓶颈**：所有 JS 与原生通信都通过 Bridge，需 JSON 序列化/反序列化，高频交互（如拖拽）有 40%+ 性能损失
2. **启动速度慢**：App 启动时加载所有 Native Modules（全量初始化）
3. **渲染异步**：setState → 异步通知 → 异步渲染，列表滚动无法达到 60 FPS

**新架构三大核心组件：**

1. **JSI（JavaScript Interface）**：JS 代码直接持有原生对象引用，无需 Bridge，直接 C++ 调用，高频交互性能提升 40%+
2. **Fabric 渲染器**：同步渲染，原生 UI 更新与 JS 更新同步执行，帧率稳定 55-60 FPS
3. **Turbo Modules**：懒加载，按需初始化，启动速度提升 30-50%

**Hermes 引擎优化：**
- JS → Hermes Bytecode（AOT 预编译），无需解析，内存占用降低 20-30%
- 优化 GC 策略，减少卡顿

**并发渲染（React 18+）：**
- `useTransition` 标记非紧急更新（如搜索输入），防止白屏卡顿
- `Suspense` 支持流式渲染

### 12.4.3 React Native 布局与样式系统

#### 知识点详解

**Flexbox 布局核心：**

```jsx
import { StyleSheet, Dimensions } from 'react-native';

const { width, height } = Dimensions.get('window');

// RN 中 Flexbox 与 CSS 几乎一致，以下是常见差异

// 1. flexDirection 默认值不同
// CSS: flex-direction: row（默认水平）
// RN: flexDirection: 'column'（默认垂直）
const styles = StyleSheet.create({
    container: {
        flex: 1,                    // 占据父容器全部剩余空间
        flexDirection: 'column',    // 默认 column，无需显式声明
        justifyContent: 'center',   // 主轴对齐
        alignItems: 'center',       // 交叉轴对齐
    },
    row: {
        flexDirection: 'row',        // 水平排列
        justifyContent: 'space-between',
        alignItems: 'center',
    },
});

// 2. position: relative 和 absolute（与 CSS 一致）
const styles = StyleSheet.create({
    overlay: {
        position: 'absolute',
        top: 0, left: 0, right: 0, bottom: 0,
        backgroundColor: 'rgba(0,0,0,0.5)',
    },
    badge: {
        position: 'absolute',
        top: -4, right: -4,
        width: 20, height: 20, borderRadius: 10,
        backgroundColor: 'red',
    },
});

// 3. flex 的使用
// flex:1 → 占据所有剩余空间（等价于 flex: 1 1 0）
// flexGrow:1 → 仅在有剩余空间时扩展（与 CSS flex-grow 一致）
const styles = StyleSheet.create({
    sidebar: { width: 80, flexShrink: 0 },
    content: { flex: 1 },
});

// 4. 百分数单位（RN 支持，但需父容器有明确尺寸）
const styles = StyleSheet.create({
    halfWidth: { width: '50%', height: '100%' },
});
```

```typescript
// store/userSlice.ts
import { createSlice, createAsyncThunk, PayloadAction } from '@reduxjs/toolkit';

interface UserState {
    name: string; token: string | null; loading: boolean; error: string | null;
}

const initialState: UserState = { name: '', token: null, loading: false, error: null };

export const fetchUserInfo = createAsyncThunk(
    'user/fetchUserInfo',
    async (userId: string, { rejectWithValue }) => {
        const res = await fetch(`/api/users/${userId}`);
        if (!res.ok) throw new Error('获取用户信息失败');
        return await res.json();
    }
);

const userSlice = createSlice({
    name: 'user', initialState,
    reducers: {
        setUser: (state, action: PayloadAction<{ name: string; token: string }>) => {
            state.name = action.payload.name; state.token = action.payload.token;
        },
        logout: (state) => { state.name = ''; state.token = null; },
    },
    extraReducers: (builder) => {
        builder
            .addCase(fetchUserInfo.pending, (state) => { state.loading = true; })
            .addCase(fetchUserInfo.fulfilled, (state, action) => {
                state.loading = false; state.name = action.payload.name;
            })
            .addCase(fetchUserInfo.rejected, (state, action) => {
                state.loading = false; state.error = action.payload as string;
            });
    },
});

export const { setUser, logout } = userSlice.actions;
export default userSlice.reducer;
```

```kotlin
// android/app/src/main/java/com/myapp/MyModule.kt
package com.myapp

import com.facebook.react.bridge.*
import android.os.Build

class MyModule(reactContext: ReactApplicationContext) :
    ReactContextBaseJavaModule(reactContext) {

    override fun getName(): String = "MyModule"

    @ReactMethod
    fun getDeviceName(promise: Promise) {
        promise.resolve(Build.MODEL)
    }

    @ReactMethod
    fun multiply(a: Double, b: Double, promise: Promise) {
        promise.resolve(a * b)
    }
}

// 注册模块
class MyPackage : ReactPackage {
    override fun createNativeModules(reactContext: ReactApplicationContext): List<NativeModule> {
        return listOf(MyModule(reactContext))
    }
}
```

```typescript
// src/native/MyModule.ts
import { NativeModules, Platform } from 'react-native';

interface MyModuleInterface {
    getDeviceName(): Promise<string>;
    multiply(a: number, b: number): Promise<number>;
}

const { MyModule } = NativeModules;

export const getDeviceName = async (): Promise<string> => {
    if (Platform.OS === 'ios') {
        return await MyModule.getDeviceName();
    }
    return await MyModule.getDeviceName(); // Android 通用
};

export const multiply = (a: number, b: number): Promise<number> => {
    return MyModule.multiply(a, b);
};
```

#### 真实面试题

**题目：什么时候需要自定义原生模块？如何在 React Native 中编写 iOS 和 Android 原生模块？**

**满分答案：**

**需要原生模块的场景：**
- RN 未提供对应能力的 API（如蓝牙 BLE、NFC）
- 需要接入第三方原生 SDK（支付、推送、人脸识别）
- 极致性能要求的场景（如复杂音视频处理）
- 现有原生代码需要复用

**iOS 编写步骤：**

1. 创建本地 CocoaPod（推荐）或直接修改 Xcode 项目
2. Swift 编写模块类，用 `@objc` 暴露方法
3. Objective-C 桥接文件（`RCT_EXTERN_MODULE` + `RCT_EXTERN_METHOD`）
4. JS 层通过 `NativeModules` 调用

**Android 编写步骤：**

1. Kotlin/Java 编写 `ReactContextBaseJavaModule` 子类
2. 用 `@ReactMethod` 注解暴露方法
3. 创建 `ReactPackage` 注册模块
4. 在 `MainApplication.kt` 中添加 Package
5. JS 层调用（与 iOS 相同接口）

**两种通信方式：**
- Promise（异步，推荐）：`await MyModule.method()` → `promise.resolve/reject`
- Callback（回调，废弃中）：`MyModule.method((err, result) => {})`

### 12.4.6 React Native iOS 打包发布完整流程

#### 知识点详解

**iOS 打包整体流程：**

```
开发调试 → 生成 JS Bundle → 嵌入 Xcode 项目 → Xcode Archive → 导出 .ipa → App Store Connect 上架
```

**Step 1：配置 package.json 构建命令：**

```json
// package.json
{
  "scripts": {
    "build-release-ios": "react-native bundle --platform ios --dev false --entry-file index.js --bundle-output ios/bundle/main.jsbundle --assets-dest ios/bundle",
    "build-debug-ios": "react-native bundle --platform ios --dev true --entry-file index.js --bundle-output ios/bundle/debug.jsbundle --assets-dest ios/bundle"
  }
}
```

**Step 2：生成 JS Bundle：**

```bash
# 创建输出目录
mkdir -p ios/bundle

# 生成 Release Bundle（生产环境）
npx react-native bundle \
  --platform ios \
  --dev false \
  --entry-file index.js \
  --bundle-output ios/bundle/main.jsbundle \
  --assets-dest ios/bundle \
  --minify true

# 或使用 npm script
npm run build-release-ios
```

生成的 `ios/bundle/` 目录包含：
- `main.jsbundle`：JS 代码打包产物（Hermes Bytecode 格式）
- `assets/`：图片、字体等静态资源

**Step 3：Xcode 项目配置：**

```bash
# 方式一：通过 Xcode GUI
# 1. 打开 ios/YourApp.xcworkspace
# 2. 将 ios/bundle/main.jsbundle 拖入 Xcode 项目
#    - 选中 "Create folder references"（蓝色文件夹图标）
#    - 确保 "Copy items if needed" 未勾选（避免重复）

# 3. 配置签名（Xcode 自动管理或手动）
# TARGETS → Signing & Capabilities → Team 选择开发者账号

# 4. 修改 AppDelegate.m 使 Bundle 离线可用
```

**AppDelegate.m 配置（离线加载 JS Bundle）：**

```objc
// iOS AppDelegate.m（旧架构）
#import <React/RCTBundleURLProvider.h>
#import <React/RCTRootView.h>

    return [[RCTBundleURLProvider sharedSettings] jsBundleURLForBundleRoot:@"index" fallbackResource:nil];

    return [[NSBundle mainBundle] URLForResource:@"main" withExtension:@"jsbundle"];

// 如果使用 Hermes
#if DEBUG
    return [[RCTBundleURLProvider sharedSettings] jsBundleURLForBundleRoot:@"index" fallbackResource:nil];
#else
    // Hermes Bytecode：.hbc 文件

#endif
```

**Step 4：Xcode Archive 打包：**

```bash
# 方式一：命令行打包
cd ios

# Archive
xcodebuild archive \
  -workspace YourApp.xcworkspace \

  CODE_SIGNING_REQUIRED=NO \
  CODE_SIGNING_ALLOWED=NO

# 导出 .ipa

  -archivePath ./build/YourApp.xcarchive \
  -exportOptionsPlist ExportOptions.plist \
  -exportPath ./build/output
```

```bash
# 方式一：Xcode 直接上传（推荐）

# 方式二：Transporter 上传 .ipa
```

```
1. 登录 App Store Connect（appstoreconnect.apple.com）
2. 我的 App → 创建 App
   - 名称、Bundle ID（与 Xcode 一致）、SKU
   - 语言、类别

```

```
- 开发证书（Development）：真机调试

描述文件（Provisioning Profile）：
- App ID：精确匹配 Bundle Identifier（例：com.company.app）

- 设备：Ad Hoc 需注册设备 UDID

常见问题：
- Bundle ID 不匹配 → Error: "No profiles for 'com.xxx' were found"
- 证书过期 → 更新证书，重新下载描述文件
- Release 打包用 Development 证书 → 只能安装到指定设备，无法上架
```

**题目：如何为 React Native 项目配置 CI/CD？CodePush 热更新的原理是什么？如何发布热更新？**

**满分答案：**

**CI/CD 配置要点：**

```yaml
# 关键步骤
1. npm ci 安装依赖（带缓存）
2. npx react-native bundle 生成 JS Bundle（iOS/Android 两个平台）
3. Xcode build（macOS runner，Apple 生态限制）
4. Gradle assembleRelease（Linux/macOS 均可）

```

**CodePush 热更新原理：**

```
CodePush 服务器（Microsoft App Center）
    ↓ 存储 JS Bundle（差分包/全量包）
用户 App 启动
    ↓ CodePush SDK 检查更新
有更新 → 下载 Bundle → 重启 App 加载新 Bundle
无更新 → 继续使用已安装的 Bundle
```

**优势：**
- JS Bundle 更新无需重新提交 App Store / Google Play 审核
- 可以向不同用户群发布不同版本（A/B 测试）
- 紧急 bug 可在几小时内修复上线

**注意事项：**
- App Store 审核可能拒绝强制热更新（需合理使用）
- 热更新不能更新原生代码变更（如新增原生模块）
- Android 11+ 需要配置 `<queries>` 允许 CodePush 服务器访问

### 12.4.9 React Native 性能优化

#### 知识点详解

**React Native 性能问题来源：**

```
JS 线程（Bridge/JSI 通信）
    ↓ 调用
原生 UI 线程（渲染、布局）
    ↓
两个线程需同步 → 帧率下降（卡顿）

主要瓶颈：
1. JS 与原生通信（Bridge 序列化）
2. 长列表渲染（无虚拟化）
3. 不必要的重渲染（props/state 未优化）
4. 图片加载（未压缩、未缓存）
5. 大 Bundle 加载（启动慢）
```

**列表优化（FlatList 关键配置）：**

```jsx
// ❌ 低性能写法：所有项全部渲染，无虚拟化
<FlatList
    data={items}
    renderItem={renderItem}
    keyExtractor={(item) => item.id}
    // ❌ 缺失关键优化
/>

// ✅ 高性能写法
<FlatList
    data={items}
    keyExtractor={(item) => item.id}
    renderItem={renderItem}

    // 虚拟化关键配置
    windowSize={21}              // 屏幕上下各渲染多少项（默认 21）
    maxToRenderPerBatch={10}      // 每批最多渲染多少项
    updateCellsBatchingPeriod={50} // 每批渲染间隔（ms）
    initialNumToRender={10}       // 初始渲染数量
    removeClippedSubviews={true}  // 屏幕外视图卸载（性能提升显著）

    // 列表为空/加载中
    ListEmptyComponent={<Loading />}
    ListFooterComponent={<EndOfList />}
    ListHeaderComponent={<Header />}

    // 下拉刷新 + 无限滚动
    onRefresh={refreshData}
    refreshing={refreshing}
    onEndReached={loadMore}
    onEndReachedThreshold={0.3}  // 距离底部 30% 时触发加载

    // 避免匿名函数在 render 中创建
    ItemSeparatorComponent={Separator}  // 非内联组件
/>

// ✅ Item 组件优化：React.memo + useCallback
const ListItem = React.memo(function ListItem({ item, onPress }) {
    console.log('Item 渲染:', item.id);
    return (
        <TouchableOpacity onPress={() => onPress(item.id)}>
            <Text>{item.title}</Text>
        </TouchableOpacity>
    );
});

// 父组件中用 useCallback 避免子组件重渲染
const renderItem = useCallback(({ item }) => (
    <ListItem item={item} onPress={handlePress} />
), [handlePress]);

const handlePress = useCallback((id: string) => {
    navigation.navigate('Detail', { id });
}, [navigation]);
```

### 12.4.10 React Native 与 Flutter / 原生开发对比

#### 知识点详解

**React Native vs Flutter vs 原生开发对比：**

| 维度 | React Native | Flutter | iOS 原生（Swift） | Android 原生（Kotlin） |
|------|-------------|---------|------------------|---------------------|
| 语言 | JavaScript/TypeScript | Dart | Swift | Kotlin |
| 渲染方式 | 原生组件 | Skia 自绘 | 系统原生 UI | 系统原生 UI |
| 性能 | 较好（接近原生） | 优秀（60-120 FPS） | 最优 | 最优 |
| 开发效率 | 高（React 生态） | 高（Dart 生态小） | 低（需双平台） | 低（需双平台） |
| 第三方库 | 丰富（npm） | 一般（pub.dev） | 丰富 | 丰富 |
| UI 一致性 | 好（Android/iOS 风格不同） | 优秀（完全自定义） | 系统原生体验 | 系统原生体验 |
| 包体积 | 中等（40-80MB） | 较大（含 Skia） | 较小 | 较小 |
| 入门门槛 | 低（前端友好） | 中（Dart 语言） | 高 | 高 |
| 学习曲线 | 陡峭（新架构/RN 特有） | 较陡 | 平缓 | 平缓 |
| 适用场景 | 互联网 App、电商、社交 | 高度自定义 UI、游戏 | 金融、游戏、高性能 | 金融、游戏、高性能 |

**React Native vs Flutter 核心区别：**

```
React Native：
JSX → React.createElement → 原生组件（UIView / View）
       ↓ 平台原生渲染（每个平台风格不同）

Flutter：
Dart 代码 → Skia 图形库自绘
       ↓ 完全自定义渲染（跨平台 UI 一致）
       性能更好，但包体积更大
```

**RN vs Flutter 场景选型：**

```
选 React Native：
选 React Native：
✅ 团队熟悉 React / JavaScript
✅ 需要快速上线 iOS + Android
✅ 主要业务逻辑为主，UI 偏传统
✅ 需要接入大量已有 npm 包
✅ 团队规模小，需复用前端经验

选 Flutter：
✅ 需要高度自定义的 UI（特殊设计系统）
✅ 对性能要求极高（如游戏、高频动画）
✅ 需要跨平台 + Web + 桌面端
✅ 团队无 React 经验，愿意学 Dart
✅ 需要像素级设计还原（设计稿一致）

选原生开发：
✅ 对性能有极端要求（大型游戏、AR/VR）
✅ 需要大量平台特有 API
✅ 长期维护的核心 App（金融、医疗）
✅ 有专职 iOS/Android 工程师团队
```

**React Native 踩坑经验（常见问题）：**

```
1. 版本升级：RN 大版本升级成本高，每次升级需测试全部功能
   → 建议：依赖固定版本，避免随意升级

2. 原生模块维护：第三方原生库未及时适配新架构
   → 建议：选择活跃维护的库，关注 RN 版本兼容性

3. 双平台差异：同一 RN 组件在不同平台表现不同
   → 建议：使用 Platform.select({ ios: ..., android: ... }) 做平台区分

4. Bridge 性能：旧架构高频交互卡顿
   → 建议：升级到 RN 0.76+（新架构默认）

5. Android 兼容性：国产 ROM（华为、小米）后台限制
   → 建议：使用 react-native-background-timer 或前台服务
```

**统一错误处理（Error Boundary）：**

```tsx
// src/components/ErrorBoundary.tsx
import React, { Component, ErrorInfo, ReactNode } from 'react';
import { View, Text, TouchableOpacity, StyleSheet } from 'react-native';

interface Props { children: ReactNode; fallback?: ReactNode; }
interface State { hasError: boolean; error: Error | null; }

export class ErrorBoundary extends Component<Props, State> {
    state: State = { hasError: false, error: null };

    static getDerivedStateFromError(error: Error): State {
        return { hasError: true, error };
    }

    componentDidCatch(error: Error, info: ErrorInfo) {
        // 上报错误到监控平台（Sentry）
        console.error('ErrorBoundary caught:', error, info.componentStack);
        // Sentry.captureException(error, { extra: { componentStack: info.componentStack } });
    }

    render() {
        if (this.state.hasError) {
            return this.props.fallback ?? (
                <View style={styles.container}>
                    <Text style={styles.title}>Something went wrong</Text>
                    <Text style={styles.error}>{this.state.error?.message}</Text>
                    <TouchableOpacity
                        style={styles.button}
                        onPress={() => this.setState({ hasError: false, error: null })}
                    >
                        <Text style={styles.buttonText}>Try Again</Text>
                    </TouchableOpacity>
                </View>
            );
        }
        return this.props.children;
    }
}

const styles = StyleSheet.create({
    container: { flex: 1, justifyContent: 'center', alignItems: 'center', padding: 20 },
    title: { fontSize: 18, fontWeight: 'bold', color: '#333', marginBottom: 8 },
    error: { fontSize: 14, color: '#999', textAlign: 'center', marginBottom: 16 },
    button: { backgroundColor: '#007AFF', paddingHorizontal: 24, paddingVertical: 12, borderRadius: 8 },
    buttonText: { color: '#fff', fontSize: 16, fontWeight: '600' },
});
```
