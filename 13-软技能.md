## 十三、软技能

### 13.1 代码审查（Code Review）

**Code Review 的目的：**

1. 发现 bug 和潜在问题
2. 确保代码质量和一致性
3. 知识共享和团队学习
4. 维护代码库的可维护性

**Code Review 清单：**

```
功能性：
□ 代码是否实现了需求？
□ 边界条件是否处理？
□ 错误处理是否完善？

代码质量：
□ 代码是否易于理解？
□ 变量和函数命名是否清晰？
□ 是否有重复代码？
□ 函数是否单一职责？

性能：
□ 是否有不必要的循环或计算？
□ 是否有内存泄漏风险？
□ 数据库查询是否优化？

安全：
□ 是否有 XSS/CSRF 风险？
□ 敏感数据是否正确处理？
□ 权限验证是否完善？

测试：
□ 是否有单元测试？
□ 测试覆盖率是否足够？
```

### 13.2 微前端

**微前端概念：**

将大型前端应用拆分为多个独立的小应用，每个应用可以独立开发、部署。

**qiankun 使用：**

```javascript
// 主应用
import { registerMicroApps, start } from 'qiankun';

registerMicroApps([
    {
        name: 'react-app',
        entry: '//localhost:7100',
        container: '#container',
        activeRule: '/react',
        props: {
            name: 'qiankun',
        }
    },
    {
        name: 'vue-app',
        entry: '//localhost:7200',
        container: '#container',
        activeRule: '/vue'
    }
]);

start();

// 子应用（React）
export function bootstrap() {
    console.log('react app bootstraped');
}

export function mount(props) {
    ReactDOM.render(<App />, props.container.querySelector('#root'));
}

export function unmount(props) {
    ReactDOM.unmountComponentAtNode(props.container.querySelector('#root'));
}
```

### 13.3 性能监控

```javascript
// Sentry 错误监控
import * as Sentry from '@sentry/react';

Sentry.init({
    dsn: 'https://xxx@sentry.io/xxx',
    environment: process.env.NODE_ENV,
    tracesSampleRate: 1.0
});

// 自定义错误上报
Sentry.captureException(new Error('Something went wrong'));
Sentry.captureMessage('Something happened');

// 用户信息
Sentry.setUser({ id: '123', email: 'user@example.com' });

// 自定义性能监控
class PerformanceMonitor {
    static report(metric) {
        fetch('/api/metrics', {
            method: 'POST',
            body: JSON.stringify(metric),
            headers: { 'Content-Type': 'application/json' }
        });
    }
    
    static init() {
        // 监控 LCP
        new PerformanceObserver((list) => {
            const entries = list.getEntries();
            const lcp = entries[entries.length - 1];
            this.report({ type: 'LCP', value: lcp.startTime });
        }).observe({ type: 'largest-contentful-paint', buffered: true });
        
        // 监控 FID
        new PerformanceObserver((list) => {
            for (const entry of list.getEntries()) {
                this.report({
                    type: 'FID',
                    value: entry.processingStart - entry.startTime
                });
            }
        }).observe({ type: 'first-input', buffered: true });
        
        // 监控 JS 错误
        window.addEventListener('error', (event) => {
            this.report({
                type: 'JS_ERROR',
                message: event.message,
                filename: event.filename,
                lineno: event.lineno
            });
        });
        
        // 监控 Promise 错误
        window.addEventListener('unhandledrejection', (event) => {
            this.report({
                type: 'PROMISE_ERROR',
                reason: event.reason?.message
            });
        });
    }
}
```

## 13.x 技术挑战：如何描述项目中最难的问题

#### 知识点详解

**STAR 法则回答框架：**

```
S - Situation（背景）：项目背景、规模、团队
T - Task（任务）：你负责的具体任务
A - Action（行动）：你采取的具体技术方案
R - Result（结果）：量化的结果和收益
```

**高分回答的关键要素：**

1. **问题要有技术深度**（不能是"接口联调慢"这种）
2. **解决过程要有思考层次**（尝试了什么、为什么选这个方案）
3. **结果要量化**（性能提升 X%、减少 X 次报错）
4. **有反思和沉淀**（从中学到了什么）

**示例回答结构（AI 视频产品场景）：**

```
背景：
  负责 AI 视频编辑器的时间轴模块，
  用户可以在时间轴上拖拽、裁剪数十个视频片段

挑战：
  时间轴有 50+ 个片段时，拖拽操作卡顿严重（帧率降到 10fps）
  原因：每次拖拽都触发全量重渲染，且有大量 DOM 操作

解决过程：
  1. 用 Chrome DevTools Performance 面板定位瓶颈
     → 发现 React reconciliation 耗时 80ms+
  2. 尝试方案一：React.memo + useMemo
     → 效果有限，因为父组件状态变化仍触发子组件渲染
  3. 方案二：将时间轴改为 Canvas 渲染
     → 绕过 DOM，直接用 Canvas 绘制所有片段
     → 拖拽时只重绘变化区域（脏矩形优化）
  4. 方案三：拖拽期间用 requestAnimationFrame 节流
     → 将更新频率锁定在 60fps

结果：
  帧率从 10fps 提升到稳定 60fps
  50+ 片段的拖拽操作流畅无卡顿

沉淀：
  复杂交互场景下，Canvas 比 DOM 更适合做高频更新的 UI
  性能优化要先测量再优化，不要凭感觉
```

**常见高质量技术挑战方向：**

```
性能类：
  - 长列表/时间轴渲染优化
  - 大文件上传/处理
  - 首屏加载优化

架构类：
  - 复杂状态管理重构
  - 微前端拆分
  - 组件库设计

AI 集成类：
  - 流式响应的 UI 处理
  - AI 结果不确定性的用户体验
  - 本地模型推理性能

工程类：
  - 构建速度优化
  - 自动化测试体系建立
  - 监控告警体系
```

#### 真实面试题

**题目：聊聊你在项目中遇到的最难的技术挑战，你是如何解决的？**

**满分答案框架：**

用 STAR 法则，重点突出：
1. **问题的复杂性**：为什么难（不是普通 bug）
2. **系统性排查**：用工具定位，不是靠猜
3. **方案对比**：尝试了多种方案，说明取舍
4. **量化结果**：数字说话
5. **经验沉淀**：从中学到了什么原则

## 13.x 职业规划与 AI 领域持续学习

#### 知识点详解

**职业规划回答框架：**

```
短期（1-2年）：
  - 在当前方向深耕，成为某个领域专家
  - 具体技术目标（如：掌握 WebGPU、深入 AI 工程化）

中期（3-5年）：
  - 技术 + 业务双向成长
  - 带领团队或主导核心项目

长期：
  - 技术专家 or 技术管理（根据自身倾向）
```

**AI 领域持续学习计划（具体化）：**

```
技术层面：
  1. 前端 AI 工程化
     - LangChain.js / Vercel AI SDK
     - Prompt Engineering 最佳实践
     - RAG（检索增强生成）前端集成

  2. 浏览器 AI 能力
     - WebGPU + ONNX Runtime Web
     - Transformers.js（浏览器端 HuggingFace）
     - MediaPipe（实时视觉 AI）

  3. AI 产品设计
     - AI UX 设计模式
     - 流式交互设计
     - 人机协作界面

学习方式：
  - 跟进 Vercel、Anthropic、OpenAI 的前端 SDK 更新
  - 参与开源项目（如 Transformers.js）
  - 做 side project 验证想法
  - 阅读 AI 产品的技术博客（Figma、Linear 等）
```

**回答示例：**

```
短期目标：
  在 AI 视频/内容生成方向深耕，
  重点掌握 WebGPU 本地推理、流式 UI 设计模式，
  目标是能独立负责 AI 功能的前端架构设计

中期目标：
  成为 AI 前端领域的技术专家，
  能够在技术选型、架构设计上给团队提供方向

持续学习计划：
  - 每周跟进 AI 前端相关的技术动态（Vercel AI SDK、WebGPU 规范）
  - 每月做一个 side project 验证新技术
  - 参与开源社区，贡献代码或文档
```

#### 真实面试题

**题目：你未来的职业规划是什么？在 AI 领域有什么持续学习的计划？**

**满分答案：**

**回答要点：**
1. **方向明确**：不要说"走一步看一步"，要有具体方向
2. **与岗位契合**：规划要和应聘公司/岗位方向一致
3. **学习计划具体**：不是"多看书"，而是具体的技术栈和学习路径
4. **展示主动性**：已经在做的事情（side project、开源贡献等）

**AI 领域学习重点：**
- 前端 AI 工程化（LangChain.js、Vercel AI SDK）
- 浏览器端 AI（WebGPU + ONNX、Transformers.js）
- AI UX 设计模式（流式交互、不确定性处理）
