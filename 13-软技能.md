## 13.1 代码审查（Code Review）

**Code Review 的目的：**

1. 发现 bug 和潜在问题
2. 确保代码质量和一致性
3. 知识共享和团队学习
4. 维护代码库的可维护性

**Code Review 清单：**

```
功能性：
□ 代码是否实现了需求？
□ 边界条件是否处理？
□ 错误处理是否完善？

代码质量：
□ 代码是否易于理解？
□ 变量和函数命名是否清晰？
□ 是否有重复代码？
□ 函数是否单一职责？

性能：
□ 是否有不必要的循环或计算？
□ 是否有内存泄漏风险？
□ 数据库查询是否优化？

安全：
□ 是否有 XSS/CSRF 风险？
□ 敏感数据是否正确处理？
□ 权限验证是否完善？

测试：
□ 是否有单元测试？
□ 测试覆盖率是否足够？
```

## 13.3 项目管理与估时

**工期估算方法：**

```
三点估算法：
- 乐观时间（O）：一切顺利的情况
- 最可能时间（M）：正常情况
- 悲观时间（P）：遇到问题的情况

期望时间 = (O + 4M + P) / 6

示例：
- 乐观：2天，最可能：3天，悲观：6天
- 期望 = (2 + 12 + 6) / 6 = 3.3天
- 实际估时：3.3天 × 1.2（缓冲系数）≈ 4天
```

**需求拆解：**

```
大任务 → 小任务（每个 < 1天）
原则：
- 可独立完成
- 可独立验证
- 有明确的完成标准

示例：
"实现用户登录功能" →
  □ 登录表单 UI（0.5天）
  □ 表单验证逻辑（0.5天）
  □ 接口联调（0.5天）
  □ Token 存储与刷新（0.5天）
  □ 错误处理与提示（0.5天）
```

## 13.5 技术挑战：如何描述项目中最难的问题

#### 知识点详解

**STAR 法则回答框架：**

```
S - Situation（背景）：项目背景、规模、团队
T - Task（任务）：你负责的具体任务
A - Action（行动）：你采取的具体技术方案
R - Result（结果）：量化的结果和收益
```

**高分回答的关键要素：**

1. **问题要有技术深度**（不能是"接口联调慢"这种）
2. **解决过程要有思考层次**（尝试了什么、为什么选这个方案）
3. **结果要量化**（性能提升 X%、减少 X 次报错）
4. **有反思和沉淀**（从中学到了什么）

**示例回答结构（AI 视频产品场景）：**

```
背景：
  负责 AI 视频编辑器的时间轴模块，
  用户可以在时间轴上拖拽、裁剪数十个视频片段

挑战：
  时间轴有 50+ 个片段时，拖拽操作卡顿严重（帧率降到 10fps）
  原因：每次拖拽都触发全量重渲染，且有大量 DOM 操作

解决过程：
  1. 用 Chrome DevTools Performance 面板定位瓶颈
     → 发现 React reconciliation 耗时 80ms+
  2. 尝试方案一：React.memo + useMemo
     → 效果有限，因为父组件状态变化仍触发子组件渲染
  3. 方案二：将时间轴改为 Canvas 渲染
     → 绕过 DOM，直接用 Canvas 绘制所有片段
     → 拖拽时只重绘变化区域（脏矩形优化）
  4. 方案三：拖拽期间用 requestAnimationFrame 节流
     → 将更新频率锁定在 60fps

结果：
  帧率从 10fps 提升到稳定 60fps
  50+ 片段的拖拽操作流畅无卡顿

沉淀：
  复杂交互场景下，Canvas 比 DOM 更适合做高频更新的 UI
  性能优化要先测量再优化，不要凭感觉
```

**常见高质量技术挑战方向：**

```
性能类：
  - 长列表/时间轴渲染优化
  - 大文件上传/处理
  - 首屏加载优化

架构类：
  - 复杂状态管理重构
  - 微前端拆分
  - 组件库设计

AI 集成类：
  - 流式响应的 UI 处理
  - AI 结果不确定性的用户体验
  - 本地模型推理性能

工程类：
  - 构建速度优化
  - 自动化测试体系建立
  - 监控告警体系
```

#### 真实面试题

**题目：聊聊你在项目中遇到的最难的技术挑战，你是如何解决的？**

**满分答案框架：**

用 STAR 法则，重点突出：
1. **问题的复杂性**：为什么难（不是普通 bug）
2. **系统性排查**：用工具定位，不是靠猜
3. **方案对比**：尝试了多种方案，说明取舍
4. **量化结果**：数字说话
5. **经验沉淀**：从中学到了什么原则

## 13.7 最难的 AI 前端技术问题

### 知识点详解

**AI 前端常见技术难点排行榜：**

```
⭐ 基础：流式渲染、Token 计算、打字机效果
⭐⭐ 中级：长对话虚拟滚动、Markdown 增量渲染
⭐⭐⭐ 高级：循环引用深拷贝、并发调度器
⭐⭐⭐⭐ 专业：React Fiber 架构、RAG 前端实现
⭐⭐⭐⭐⭐ 专家：Agent 工作流可视化、TTFT 优化
```

### 真实面试题

**题目：聊聊你在过去一年中遇到的最难的 AI 前端技术问题，你是如何解决的？**

**满分答案（STAR 法则）：**

**S - Situation（背景）：**
> 在做一个企业级 AI 对话助手时，遇到长对话（100+ 条消息）页面滚动卡顿，FPS 跌到 10 以下，用户体验极差。

**T - Task（任务）：**
> 将长对话页面的滚动 FPS 从 10 提升到 55+，同时保持消息列表、Markdown、图片等功能完整。

**A - Action（行动）：**

**1. 精准定位瓶颈**
```typescript
// Chrome DevTools Performance 面板分析
// 发现：
// - DOM 节点数 > 50000
// - React 重渲染耗时 300ms+
// - 每条消息都在重渲染（即使内容没变）
```

**2. 虚拟滚动**
```tsx
import { Virtuoso } from 'react-virtuoso';
<Virtuoso
    data={messages}
    followOutput="smooth"
    itemContent={(i, msg) => <MessageItem key={msg.id} {...msg} />}
/>
```

**3. 组件优化**
```tsx
const MessageItem = memo(({ msg }: { msg: Message }) => {
    return (
        <div className="message">
            <MarkdownView content={msg.content} />
        </div>
    );
}, (prev, next) =>
    prev.msg.id === next.msg.id &&
    prev.msg.content === next.msg.content
);
```

**4. 流式渲染防抖**
```typescript
// 流式更新每 16ms 更新一次（~60fps）
const debouncedSetMessages = useMemo(
    () => debounce(setMessages, 16),
    []
);
```

**R - Result（结果）：**
> 长对话（500 条消息）滚动 FPS 从 10 提升到 58，内存占用减少 60%，用户投诉率下降 80%。

**经验教训（加分项）：**
> 性能优化必须先测量再优化。找准瓶颈点比写更多代码有效 10 倍。

**题目2：聊聊你在过去一年中遇到的最难的AI前端技术问题，你是如何解决的？**

**满分答案（示例）：**

**问题：** AI聊天长文本（10万+字符）渲染严重卡顿

**分析：**
- 直接原因：DOM节点过多，重排重绘频繁
- 根本原因：一次性插入大量内容，浏览器主线程阻塞

**解决方案：**
1. **虚拟滚动**：只渲染可视区域内容
2. **分段渲染**：每100字符批量插入，减少DOM操作
3. **requestAnimationFrame**：利用浏览器空闲时间渲染
4. **WebWorker**：Token计算放到后台线程

**结果：**
- 渲染帧率从5fps提升到60fps
- 支持无限长度对话
- 内存占用降低70%

## 13.11 多人协作 AI 项目的前后端协作规范

#### 知识点详解

**协作规范的核心：接口约定 + 文档先行**

**一、接口协议规范：**

```typescript
// AI 接口统一格式（前端后端共享类型）
// types/ai.ts
interface AIRequest {
  model: string;
  messages: Array<{
    role: 'user' | 'assistant' | 'system';
    content: string;
  }>;
  temperature?: number;
  max_tokens?: number;
  stream?: boolean;
}

interface AIResponse {
  id: string;
  model: string;
  choices: Array<{
    message: { role: string; content: string };
    finish_reason: string;
  }>;
  usage: {
    prompt_tokens: number;
    completion_tokens: number;
    total_tokens: number;
  };
}

// 流式响应格式（NDJSON，每行一个JSON对象）
// {"type":"chunk","delta":"Hello"}
// {"type":"chunk","delta":" world"}
// {"type":"done","usage":{...}}
```

**二、前后端接口约定：**

```typescript
// 约定：状态码约定
// 200: 成功（流式结束时返回最终 usage）
// 400: 参数错误（invalid_request）
// 401: 鉴权失败
// 403: 内容审查拦截（content_filter）
// 429: 请求限流（rate_limit），响应头 Retry-After
// 500: 服务端错误
// 504: 模型超时（gateway_timeout）

// 约定：超时应由网关返回504，前端 AbortController 兜底
```

**三、前端与算法/模型组协作：**

| 协作点 | 前端职责 | 模型组职责 |
|--------|---------|-----------|
| Prompt 版本 | 前端管理，可热更新 | 提供默认 Prompt |
| 模型选型 | 前端可切换模型 | 提供模型列表和能力说明 |
| Token 预算 | 前端计算，提示超限 | 告知上下文限制 |
| 输出格式 | 前端解析（Markdown/JSON） | 统一约定格式 |
| 异常处理 | 前端捕获显示 | 返回结构化错误码 |

**四、Git 工作流：**

```bash
# 分支命名
feat/ai-chat-streaming    # AI 功能开发
feat/ai-prompt-v2         # Prompt 版本迭代
fix/ai-timeout            # AI 异常修复

# PR 规范
# AI 功能 PR 需要包含：
# 1. AI 效果截图/录屏
# 2. Token 消耗对比（优化类 PR）
# 3. Prompt 变更说明（如果涉及）
```

#### 真实面试题

**题目：在多人协作的 AI 项目中，如何制定前端与后端、模型组之间的协作规范？**

**满分答案：**

**1. 接口规范先行（最重要）**
- 用 OpenAPI/Swagger 定义 AI 接口，双方必须严格遵守
- AI 请求/响应结构用共享 TypeScript 类型包（`@/types/ai`），前后端共享
- 流式响应格式用 NDJSON，格式变更必须通知前端

**2. Prompt 版本管理**
- Prompt 放在配置文件或数据库，前端可热更新，无需发版
- Prompt 变更必须记录：改了什么、为什么改、预期效果
- 重要场景的 Prompt 变更需要 A/B 测试

**3. 异常处理约定**
- 统一错误码体系：前端根据错误码显示对应提示
- 后端返回结构化错误：`{ code, message, detail }`
- 模型组提供 Token 上限/速率限制，前端做预防性计算

**4. 监控与反馈闭环**
- 前端埋点采集 AI 输出质量（用户反馈 / 点赞点踩）
- 模型组根据反馈迭代 Prompt / 模型
- 前端接收新版本，快速 A/B 测试验证效果

**面试加分点：**
- 提到"用 tRPC / Protobuf 共享类型，避免前后端类型不一致"
- 提到"Prompt 变更必须记录，避免线上问题无法回溯"
- 提到"AI 功能测试困难，用 Playwright 做自动化验收测试"
