## 13.0 软技能概述

## 13.1 代码审查（Code Review）

**Code Review 的目的：**

1. 发现 bug 和潜在问题
2. 确保代码质量和一致性
3. 知识共享和团队学习
4. 维护代码库的可维护性

**Code Review 清单：**

```
功能性：
□ 代码是否实现了需求？
□ 边界条件是否处理？
□ 错误处理是否完善？

代码质量：
□ 代码是否易于理解？
□ 变量和函数命名是否清晰？
□ 是否有重复代码？
□ 函数是否单一职责？

性能：
□ 是否有不必要的循环或计算？
□ 是否有内存泄漏风险？
□ 数据库查询是否优化？

安全：
□ 是否有 XSS/CSRF 风险？
□ 敏感数据是否正确处理？
□ 权限验证是否完善？

测试：
□ 是否有单元测试？
□ 测试覆盖率是否足够？
```


## 13.2 沟通与协作

**向上沟通（与 Leader/PM）：**

```
原则：
1. 结论先行：先说结论，再说原因
2. 量化表达：用数据说话
3. 提供方案：不只提问题，同时给出解决方案

示例：
❌ "这个需求做不了"
✅ "这个需求按当前方案需要 3 周，如果简化 X 功能可以缩短到 1 周，
   或者我们可以分两期：先上核心功能，再迭代完整版"
```

**技术方案评审：**

```
评审文档结构：
1. 背景与目标（为什么做）
2. 方案设计（怎么做）
   - 技术选型及理由
   - 核心流程图/架构图
   - 关键接口设计
3. 风险与应对
4. 工期估算
5. 验收标准
```

**跨团队协作：**

```
前后端协作：
- 接口文档先行（Swagger/Apifox）
- 约定数据格式和错误码规范
- Mock 数据并行开发

设计协作：
- 设计稿标注规范（Figma）
- 组件复用约定
- 响应式断点统一
```

## 13.3 项目管理与估时

**工期估算方法：**

```
三点估算法：
- 乐观时间（O）：一切顺利的情况
- 最可能时间（M）：正常情况
- 悲观时间（P）：遇到问题的情况

期望时间 = (O + 4M + P) / 6

示例：
- 乐观：2天，最可能：3天，悲观：6天
- 期望 = (2 + 12 + 6) / 6 = 3.3天
- 实际估时：3.3天 × 1.2（缓冲系数）≈ 4天
```

**需求拆解：**

```
大任务 → 小任务（每个 < 1天）
原则：
- 可独立完成
- 可独立验证
- 有明确的完成标准

示例：
"实现用户登录功能" →
  □ 登录表单 UI（0.5天）
  □ 表单验证逻辑（0.5天）
  □ 接口联调（0.5天）
  □ Token 存储与刷新（0.5天）
  □ 错误处理与提示（0.5天）
```

## 13.4 技术分享与文档

**技术分享结构：**

```
1. 背景（为什么要分享这个）
2. 核心概念（是什么）
3. 实际案例（怎么用）
4. 踩坑经验（注意什么）
5. 总结与延伸

时间控制：
- 15分钟分享：1个核心知识点
- 30分钟分享：1个完整主题
- 60分钟分享：系统性知识体系
```

**技术文档写作：**

```
好文档的标准：
- 目标读者明确
- 结构清晰（目录、标题层级）
- 示例代码可运行
- 及时更新

常见文档类型：
- README：项目介绍、快速开始
- API 文档：接口说明
- 架构文档：系统设计决策
- 复盘文档：事故/项目复盘
```

## 13.5 技术挑战：如何描述项目中最难的问题

#### 知识点详解

**STAR 法则回答框架：**

```
S - Situation（背景）：项目背景、规模、团队
T - Task（任务）：你负责的具体任务
A - Action（行动）：你采取的具体技术方案
R - Result（结果）：量化的结果和收益
```

**高分回答的关键要素：**

1. **问题要有技术深度**（不能是"接口联调慢"这种）
2. **解决过程要有思考层次**（尝试了什么、为什么选这个方案）
3. **结果要量化**（性能提升 X%、减少 X 次报错）
4. **有反思和沉淀**（从中学到了什么）

**示例回答结构（AI 视频产品场景）：**

```
背景：
  负责 AI 视频编辑器的时间轴模块，
  用户可以在时间轴上拖拽、裁剪数十个视频片段

挑战：
  时间轴有 50+ 个片段时，拖拽操作卡顿严重（帧率降到 10fps）
  原因：每次拖拽都触发全量重渲染，且有大量 DOM 操作

解决过程：
  1. 用 Chrome DevTools Performance 面板定位瓶颈
     → 发现 React reconciliation 耗时 80ms+
  2. 尝试方案一：React.memo + useMemo
     → 效果有限，因为父组件状态变化仍触发子组件渲染
  3. 方案二：将时间轴改为 Canvas 渲染
     → 绕过 DOM，直接用 Canvas 绘制所有片段
     → 拖拽时只重绘变化区域（脏矩形优化）
  4. 方案三：拖拽期间用 requestAnimationFrame 节流
     → 将更新频率锁定在 60fps

结果：
  帧率从 10fps 提升到稳定 60fps
  50+ 片段的拖拽操作流畅无卡顿

沉淀：
  复杂交互场景下，Canvas 比 DOM 更适合做高频更新的 UI
  性能优化要先测量再优化，不要凭感觉
```

**常见高质量技术挑战方向：**

```
性能类：
  - 长列表/时间轴渲染优化
  - 大文件上传/处理
  - 首屏加载优化

架构类：
  - 复杂状态管理重构
  - 微前端拆分
  - 组件库设计

AI 集成类：
  - 流式响应的 UI 处理
  - AI 结果不确定性的用户体验
  - 本地模型推理性能

工程类：
  - 构建速度优化
  - 自动化测试体系建立
  - 监控告警体系
```

#### 真实面试题

**题目：聊聊你在项目中遇到的最难的技术挑战，你是如何解决的？**

**满分答案框架：**

用 STAR 法则，重点突出：
1. **问题的复杂性**：为什么难（不是普通 bug）
2. **系统性排查**：用工具定位，不是靠猜
3. **方案对比**：尝试了多种方案，说明取舍
4. **量化结果**：数字说话
5. **经验沉淀**：从中学到了什么原则

## 13.6 职业规划与 AI 领域持续学习

#### 知识点详解

**职业规划回答框架：**

```
短期（1-2年）：
  - 在当前方向深耕，成为某个领域专家
  - 具体技术目标（如：掌握 WebGPU、深入 AI 工程化）

中期（3-5年）：
  - 技术 + 业务双向成长
  - 带领团队或主导核心项目

长期：
  - 技术专家 or 技术管理（根据自身倾向）
```

**AI 领域持续学习计划（具体化）：**

```
技术层面：
  1. 前端 AI 工程化
     - LangChain.js / Vercel AI SDK
     - Prompt Engineering 最佳实践
     - RAG（检索增强生成）前端集成

  2. 浏览器 AI 能力
     - WebGPU + ONNX Runtime Web
     - Transformers.js（浏览器端 HuggingFace）
     - MediaPipe（实时视觉 AI）

  3. AI 产品设计
     - AI UX 设计模式
     - 流式交互设计
     - 人机协作界面

学习方式：
  - 跟进 Vercel、Anthropic、OpenAI 的前端 SDK 更新
  - 参与开源项目（如 Transformers.js）
  - 做 side project 验证想法
  - 阅读 AI 产品的技术博客（Figma、Linear 等）
```

**回答示例：**

```
短期目标：
  在 AI 视频/内容生成方向深耕，
  重点掌握 WebGPU 本地推理、流式 UI 设计模式，
  目标是能独立负责 AI 功能的前端架构设计

中期目标：
  成为 AI 前端领域的技术专家，
  能够在技术选型、架构设计上给团队提供方向

持续学习计划：
  - 每周跟进 AI 前端相关的技术动态（Vercel AI SDK、WebGPU 规范）
  - 每月做一个 side project 验证新技术
  - 参与开源社区，贡献代码或文档
```

#### 真实面试题

**题目：你未来的职业规划是什么？在 AI 领域有什么持续学习的计划？**

**满分答案：**

**回答要点：**
1. **方向明确**：不要说"走一步看一步"，要有具体方向
2. **与岗位契合**：规划要和应聘公司/岗位方向一致
3. **学习计划具体**：不是"多看书"，而是具体的技术栈和学习路径
4. **展示主动性**：已经在做的事情（side project、开源贡献等）

**AI 领域学习重点：**
- 前端 AI 工程化（LangChain.js、Vercel AI SDK）
- 浏览器端 AI（WebGPU + ONNX、Transformers.js）
- AI UX 设计模式（流式交互、不确定性处理）
