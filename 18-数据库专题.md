## 18.1 PostgreSQL 核心基础

### 18.1.1 PostgreSQL 简介与特点

#### 知识点详解

**PostgreSQL 是什么：**

PostgreSQL（俗称"Postgres"）是一个功能强大的**开源对象关系型数据库系统（ORDBMS）**，起源于 UC Berkeley 的 Ingres 项目，1989 年发布第一个版本。经过 35+ 年的持续开发，是目前功能最完善的开源数据库。

**为什么选择 PostgreSQL：**

| 特性 | 说明 |
|------|------|
| 全功能 | 支持 SQL 标准完整、功能丰富 |
| 扩展性 | 支持自定义类型、函数、索引方法 |
| 标准兼容 | 高度兼容 SQL 标准（SQL:2023） |
| 并发控制 | MVCC + ACID，完全支持事务 |
| JSON 支持 | 原生 JSON/JSONB，无需 NoSQL |
| 地理信息 | PostGIS，世界级 GIS 支持 |
| 全文搜索 | 内置全文索引，无需 Elasticsearch |
| 稳定性 | 可靠性和数据完整性业界公认 |

**PostgreSQL vs MySQL vs MongoDB 快速对比：**

| 维度 | PostgreSQL | MySQL | MongoDB |
|------|-----------|-------|---------|
| 类型 | 关系型 | 关系型 | 文档型 NoSQL |
| SQL 支持 | 完整 | 基本 | 聚合管道 |
| 事务 | ACID 完整 | InnoDB 完整 | 单文档 ACID |
| JSON | JSONB 高性能 | 支持但不如 PG | 原生 |
| 扩展性 | 通过扩展实现 | 插件较少 | 分片原生支持 |
| 适用场景 | 复杂业务、数据一致性 | Web/轻量级 | 灵活schema、大数据 |

#### 真实面试题

**题目：PostgreSQL 有哪些特点？为什么在大中型项目中越来越受欢迎？**

**满分答案：**

**核心特点：**

1. **功能最完整**：完整支持 SQL 标准，支持存储过程、触发器、视图、自定义函数、复杂查询
2. **MVCC 机制**：多版本并发控制，读不阻塞写，写不阻塞读，并发性能优秀
3. **扩展性强**：`CREATE EXTENSION` 安装扩展（PostGIS、pgvector、pgcrypto 等）
4. **JSON 原生支持**：JSONB 类型高性能存储和索引，适合混合负载
5. **高可靠**：完全 ACID 事务，支持 WAL 日志复制，主从/多主/集群方案成熟

**受欢迎原因：**
- 云原生时代：PostgreSQL 是所有主流云厂商的标准选项（Aurora PostgreSQL、RDS PostgreSQL、Azure DB for PostgreSQL）
- 统一数据平台：一个数据库满足 OLTP + 简单 OLAP + JSON + 向量搜索
- AI/向量时代：`pgvector` 扩展让 PG 支持向量数据库能力，RAG 应用首选

### 18.1.2 PostgreSQL 数据类型

#### 知识点详解

**数值类型：**

```sql
-- 整数
SMALLINT      -- 2字节，-32768~32767
INTEGER/INT   -- 4字节（默认），约21亿
BIGINT        -- 8字节，超大数字

-- 精确小数（适合货币）
DECIMAL(p,s)  -- 精确计算，p=总位数，s=小数位
NUMERIC(p,s)  -- DECIMAL 的别名

-- 近似小数（性能高但不精确）
REAL          -- 4字节，单精度
DOUBLE PRECISION -- 8字节，双精度

-- 序列
SERIAL        -- INTEGER 自增（等效 BIGSERIAL）
BIGSERIAL     -- BIGINT 自增
```

**字符类型：**

```sql
CHAR(n)       -- 定长，不足补空格
VARCHAR(n)    -- 变长，最大 n 字符（推荐）
TEXT          -- 变长，无限长度（PostgreSQL 最常用）
```

**日期时间类型：**

```sql
DATE            -- 日期（2026-04-03）
TIME            -- 时间（不含时区）
TIMESTAMP       -- 日期+时间（不含时区）
TIMESTAMPTZ     -- 日期+时间（含时区，推荐）
INTERVAL         -- 时间间隔

-- 常用操作
SELECT NOW();                    -- 当前时间戳（含时区）
SELECT CURRENT_DATE;             -- 当前日期
SELECT CURRENT_TIME;            -- 当前时间
SELECT NOW() + INTERVAL '7 days'; -- 7天后
```

**布尔/枚举/几何类型：**

```sql
-- 布尔
BOOLEAN        -- true / false / null

-- 枚举
CREATE TYPE mood AS ENUM ('sad', 'ok', 'happy');
CREATE TABLE person (name TEXT, mood mood);

-- 几何类型（PostgreSQL 独有）
POINT          -- 点 (x, y)
LINE           -- 直线
LSEG          -- 线段
BOX           -- 矩形
PATH          -- 路径
POLYGON       -- 多边形
CIRCLE        -- 圆
```

**JSON 类型（重点）：**

```sql
-- JSON：存储为文本，保存格式
-- JSONB：存储为二进制，可建索引（推荐）
CREATE TABLE api_logs (
    id SERIAL PRIMARY KEY,
    payload JSONB,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- JSONB 操作
SELECT payload->>'name' FROM api_logs;     -- 获取字符串
SELECT payload->'settings'->>'theme' FROM api_logs;  -- 嵌套
SELECT payload @> '{"name":"Alice"}' FROM api_logs;  -- 包含判断（可建索引）

-- JSONB 索引（GIN 索引）
CREATE INDEX idx_payload ON api_logs USING GIN (payload);
CREATE INDEX idx_payload_name ON api_logs ((payload->>'name'));

-- JSONB 路径查询
SELECT * FROM api_logs
WHERE payload @> '{"settings":{"theme":"dark"}}';
```

**数组类型（PostgreSQL 独有）：**

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    tags TEXT[],
    prices DECIMAL(10,2)[]
);

INSERT INTO products (tags, prices)
VALUES (ARRAY['electronics', 'sale'], ARRAY[99.99, 79.99]);

SELECT * FROM products WHERE 'electronics' = ANY(tags);
```

**范围类型：**

```sql
-- 时间范围（用于排班/预约场景）
CREATE TABLE reservations (
    room TEXT,
    period TSTZRANGE
);

SELECT * FROM reservations
WHERE period && '[2026-04-03 14:00, 2026-04-03 16:00)';
```

#### 真实面试题

**题目：PostgreSQL 有哪些常用数据类型？JSONB 和 JSON 有什么区别？**

**满分答案：**

**常用数据类型：**

| 类别 | 类型 | 用途 |
|------|------|------|
| 整数 | INT, BIGINT | ID、计数 |
| 小数 | DECIMAL, NUMERIC | 货币（精确） |
| 浮点 | REAL, DOUBLE | 科学计算 |
| 字符串 | TEXT, VARCHAR | 文本 |
| 日期 | DATE, TIMESTAMPTZ | 日期时间 |
| 布尔 | BOOLEAN | true/false |
| JSON | JSON, JSONB | 半结构化数据 |
| 数组 | TEXT[], INT[] | 多值字段 |
| UUID | UUID | 全局唯一ID |

**JSON vs JSONB：**

| 特性 | JSON | JSONB |
|------|------|-------|
| 存储格式 | 文本（保留格式） | 二进制 |

| 索引支持 | ❌ | ✅ GIN 索引 |
| 磁盘占用 | 较小 | 稍大 |
| 保留顺序 | ✅ | ❌ 不保证 |

**结论：几乎总是用 JSONB**（除了需要保留原始 JSON 格式的场景）

### 18.1.3 SQL 基础与高级查询

#### 知识点详解

**DDL（数据定义）：**

```sql
-- 创建表
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    age INTEGER CHECK (age >= 0 AND age <= 150),
    status TEXT DEFAULT 'active',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ
);

-- 修改表结构
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
ALTER TABLE users DROP COLUMN phone;
ALTER TABLE users ALTER COLUMN name TYPE VARCHAR(200);
ALTER TABLE users ADD CONSTRAINT email_unique UNIQUE (email);

-- 删除表
DROP TABLE IF EXISTS users CASCADE;
```

**DML（数据操作）：**

```sql
-- 插入
INSERT INTO users (name, email, age) VALUES ('Alice', 'alice@example.com', 25);
INSERT INTO users (name, email, age) VALUES
    ('Bob', 'bob@example.com', 30),
    ('Carol', 'carol@example.com', 28);

-- 更新
UPDATE users SET age = age + 1 WHERE id = 1;
UPDATE users SET (name, age) = ('Alice Wang', 26) WHERE id = 1;

-- 删除
DELETE FROM users WHERE id = 10;
TRUNCATE TABLE users RESTART IDENTITY CASCADE;  -- 清空表

-- Upsert（冲突处理）
INSERT INTO users (id, name, email)
VALUES (1, 'Alice', 'alice@example.com')
ON CONFLICT (id) DO UPDATE SET name = EXCLUDED.name;
```

**DQL（高级查询）：**

```sql
-- CTE（公用表表达式，WITH 子句）
WITH active_users AS (
    SELECT id, name FROM users WHERE status = 'active'
),
user_orders AS (
    SELECT user_id, COUNT(*) as order_count
    FROM orders
    GROUP BY user_id
)
SELECT u.name, o.order_count
FROM active_users u
LEFT JOIN user_orders o ON u.id = o.user_id;

-- 递归 CTE（树形结构查询）
WITH RECURSIVE tree AS (
    -- 起始条件：根节点
    SELECT id, name, parent_id, 1 as depth
    FROM categories WHERE parent_id IS NULL

    UNION ALL

    -- 递归：子节点
    SELECT c.id, c.name, c.parent_id, t.depth + 1
    FROM categories c
    JOIN tree t ON c.parent_id = t.id
)
SELECT * FROM tree ORDER BY depth, name;

-- 窗口函数（高级分析）
SELECT
    name,
    department,
    salary,
    AVG(salary) OVER (PARTITION BY department) as dept_avg,
    salary - AVG(salary) OVER (PARTITION BY department) as diff_from_avg,
    RANK() OVER (ORDER BY salary DESC) as salary_rank,
    LAG(salary, 1) OVER (ORDER BY salary DESC) as prev_salary,
    SUM(salary) OVER (ORDER BY salary ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) as running_total
FROM employees;
```

#### 真实面试题

**题目：PostgreSQL 中 CTE（WITH 子句）和窗口函数有什么用？请举例说明。**

**满分答案：**

**CTE（公用表表达式）：**

将复杂查询拆解为多个简单部分，提高可读性和可维护性：

```sql
-- 场景：查询订单数超过5的用户及其消费总额
WITH user_stats AS (
    SELECT user_id, COUNT(*) as order_count, SUM(amount) as total
    FROM orders
    GROUP BY user_id
)
SELECT u.name, s.order_count, s.total
FROM users u
JOIN user_stats s ON u.id = s.user_id
WHERE s.order_count > 5;
```

**递归 CTE（树形查询）：**

```sql
-- 场景：查询公司组织架构（员工→经理→总监→CTO）
WITH RECURSIVE org_tree AS (
    SELECT id, name, manager_id, 1 as level
    FROM employees WHERE manager_id IS NULL  -- CEO

    UNION ALL

    SELECT e.id, e.name, e.manager_id, t.level + 1
    FROM employees e
    JOIN org_tree t ON e.manager_id = t.id
)
SELECT * FROM org_tree ORDER BY level, name;
```

**窗口函数（不分组聚合）：**

```sql
-- 场景：每个部门内按工资排名，同时计算部门平均
SELECT
    name,
    department,
    salary,
    ROUND(AVG(salary) OVER (PARTITION BY department), 2) as dept_avg,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dept_rank
FROM employees;

-- 结果：按部门分组后，还能显示组内排名和组平均（无需 GROUP BY）
```

### 18.1.4 约束（Constraints）

#### 知识点详解

**六大约束类型：**

```sql
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,

    -- 非空约束
    user_id BIGINT NOT NULL,

    -- 唯一约束（可命名）
    order_no VARCHAR(50) CONSTRAINT order_no_unique UNIQUE,

    -- 主键约束（自动NOT NULL + UNIQUE）
    -- 可组合主键
    CONSTRAINT order_pk PRIMARY KEY (id),

    -- 外键约束
    CONSTRAINT fk_user FOREIGN KEY (user_id)
        REFERENCES users(id)
        ON DELETE CASCADE  -- 级联删除
        ON UPDATE CASCADE, -- 级联更新

    -- 检查约束
    amount DECIMAL(10,2) CHECK (amount >= 0),
    status VARCHAR(20) CHECK (status IN ('pending','paid','shipped','completed')),

    -- 排除约束（PostgreSQL 独有，保证不重叠）
    period TSTZRANGE,
    CONSTRAINT no_overlap EXCLUDE USING GIST (
        period WITH &&
    )
);
```

**约束管理：**

```sql
-- 添加约束
ALTER TABLE orders ADD CONSTRAINT amount_positive CHECK (amount >= 0);

-- 删除约束
ALTER TABLE orders DROP CONSTRAINT amount_positive;

-- 约束延迟检查（适合批量插入）
SET CONSTRAINTS ALL DEFERRED;
INSERT INTO orders ...;  -- 提交时才检查约束
SET CONSTRAINTS ALL IMMEDIATE;
```

#### 真实面试题

**题目：数据库有哪些约束类型？PostgreSQL 的 EXCLUDE 约束有什么用？**

**满分答案：**

**六种约束类型：**

| 约束 | 作用 | 语法 |
|------|------|------|
| NOT NULL | 不允许空值 | `name VARCHAR NOT NULL` |
| UNIQUE | 值唯一 | `email VARCHAR UNIQUE` |
| PRIMARY KEY | 主键（唯一+非空） | `id BIGINT PRIMARY KEY` |
| FOREIGN KEY | 外键关联 | `REFERENCES users(id)` |
| CHECK | 自定义条件 | `CHECK (age >= 0)` |
| EXCLUDE | 排他约束（PG独有） | `USING GIST (range WITH &&)` |

**EXCLUDE 约束的应用场景：**

防止时间重叠（如会议室预约）：

```sql
CREATE TABLE room_bookings (
    id SERIAL PRIMARY KEY,
    room_id INTEGER,
    period TSTZRANGE,
    CONSTRAINT no_double_booking EXCLUDE USING GIST (
        room_id WITH =,
        period WITH &&
    )
);

-- 插入第一笔预约：成功
INSERT INTO room_bookings (room_id, period)
VALUES (1, '[2026-04-03 09:00, 2026-04-03 10:00)');

-- 插入重叠预约：失败（PostgreSQL 报错）
INSERT INTO room_bookings (room_id, period)
VALUES (1, '[2026-04-03 09:30, 2026-04-03 10:30)');
-- ERROR: conflicting key value violates exclusion constraint "no_double_booking"
```

### 18.1.5 索引（Indexes）

#### 知识点详解

**PostgreSQL 索引类型：**

```sql
-- 1. B-Tree 索引（默认，最常用）
CREATE INDEX idx_users_email ON users (email);
CREATE INDEX idx_orders_user_date ON orders (user_id, created_at DESC);  -- 组合索引

-- 2. Hash 索引（等值查询，极快，但不适用于范围查询）
CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

-- 3. GIN 索引（倒排索引，适合数组和 JSONB）
CREATE INDEX idx_products_tags ON products USING GIN (tags);
CREATE INDEX idx_api_logs ON api_logs USING GIN (payload);

-- 4. GiST 索引（几何/地理数据，全文搜索）
CREATE INDEX idx_locations ON locations USING GIST (geom);

-- 5. BRIN 索引（块范围索引，适合时序数据，极小空间）
CREATE INDEX idx_logs_created ON logs USING BRIN (created_at);

-- 6. 部分索引（只索引满足条件的行）
CREATE INDEX idx_active_users ON users (last_login)
WHERE status = 'active';

-- 7. 表达式索引（对表达式/函数结果建索引）
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
CREATE INDEX idx_users_age_group ON users ((age / 10) * 10);

-- 唯一索引
CREATE UNIQUE INDEX idx_users_email ON users (email);
```

**索引管理与分析：**

```sql
-- 查看索引
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'users';

-- 查看查询计划（是否使用索引）
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'alice@example.com';
-- 结果：Index Scan using idx_users_email on users

-- 查看索引使用统计
SELECT
    schemaname, tablename, indexname,
    idx_scan, idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY idx_scan DESC;

-- 删除未使用的索引
SELECT schemaname || '.' || tablename as table,
       indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0 AND NOT indexname LIKE '%_pkey'
ORDER BY pg_relation_size(indexrelid) DESC;

-- 重建索引（碎片整理）
REINDEX INDEX CONCURRENTLY idx_users_email;
```

**索引选择原则：**

| 场景 | 推荐索引类型 |
|------|------------|
| 等值查询（=）| B-Tree 或 Hash |
| 范围查询（BETWEEN、>、<）| B-Tree |
| 全文搜索 | GIN + tsvector |
| JSONB 查询 | GIN |
| 地理数据 | GiST |
| 时序数据（按时间范围）| BRIN |
| 数组包含查询 | GIN |

#### 真实面试题

**题目：PostgreSQL 有哪些索引类型？如何选择？什么情况下索引会失效？**

**满分答案：**

**索引类型对比：**

| 类型 | 原理 | 适用场景 |
|------|------|---------|
| B-Tree（默认） | 平衡树 | 等值+范围+排序 |
| Hash | 哈希表 | 仅等值查询 |
| GIN | 倒排索引 | 数组、JSONB、全文 |
| GiST | 通用搜索树 | 几何、地理、范围 |
| BRIN | 块范围索引 | 时序大表（极小空间） |
| 部分索引 | 过滤条件 | 特定行子集 |

**如何选择索引：**

```sql
-- 等值查询：默认 B-Tree
CREATE INDEX idx1 ON orders (user_id);

-- 组合索引：最左前缀原则
CREATE INDEX idx2 ON orders (user_id, status, created_at);
-- 可用于：(user_id), (user_id, status), (user_id, status, created_at)
-- 不可用于：(status), (created_at)

-- JSONB 全文搜索
CREATE INDEX idx3 ON api_logs USING GIN (payload);
SELECT * FROM api_logs WHERE payload @> '{"type":"error"}';
```

**索引失效的常见情况：**

```sql
-- ❌ 使用函数/计算：索引建立在列上，不在表达式上
CREATE INDEX idx_age ON users (age);
SELECT * FROM users WHERE age + 1 > 30;  -- 失效！
-- 解决：用表达式索引
CREATE INDEX idx_age2 ON users ((age + 1));

-- ❌ 类型转换
CREATE INDEX idx_email ON users (email);
SELECT * FROM users WHERE email = 123;  -- 隐式转换，失效

-- ❌ 使用 NOT IN / <>
SELECT * FROM users WHERE email NOT IN ('a@b.com', 'c@d.com');  -- 索引通常失效

-- ❌ 前缀通配符
SELECT * FROM users WHERE email LIKE '%@gmail.com';  -- ❌
SELECT * FROM users WHERE email LIKE 'admin%';      -- ✅ 可用 B-Tree

-- ❌ 统计信息不准确（ANALYZE）
-- 解决：定期执行 ANALYZE 或 VACUUM ANALYZE

-- ✅ 优化技巧：覆盖索引（避免回表）
CREATE INDEX idx_cover ON orders (user_id) INCLUDE (amount, status);
-- 查询 SELECT amount, status FROM orders WHERE user_id = 1;
-- 直接从索引返回值，无需回表
```

## 18.2 PostgreSQL 高级特性

### 18.2.1 MVCC 与事务隔离级别

#### 知识点详解

**MVCC（多版本并发控制）：**

PostgreSQL 使用 MVCC 实现高并发，每个事务看到的是数据库的"快照"，而非实时数据。

```
事务 A（READ COMMITTED）：
T1: BEGIN;
T2: UPDATE users SET age = 30 WHERE id = 1;  -- 锁定行，获得新版本
T3: SELECT * FROM users WHERE id = 1;          -- 看到新值 30
T4: COMMIT;

事务 B（同一时间）：
T1: BEGIN;
T2: SELECT * FROM users WHERE id = 1;         -- 看到旧值 25（快照）
T3: COMMIT;                                    -- 看到 UPDATE 后的值
```

**PostgreSQL 的 VACUUM 机制：**

```sql
-- PostgreSQL UPDATE/DELETE 不立即删除旧版本
-- VACUUM 清理废弃行版本
VACUUM;                          -- 清理当前数据库
VACUUM FULL users;               -- 重建表（表锁，生产禁用）
VACUUM ANALYZE users;            -- 清理 + 更新统计信息

-- 自动 vacuum（默认开启）
ALTER TABLE users SET (autovacuum_vacuum_threshold = 50);

-- 查看表膨胀情况
SELECT
    schemaname, tablename,
    n_dead_tup, n_live_tup,
    round(n_dead_tup * 100.0 / nullif(n_live_tup + n_dead_tup, 0), 2) as dead_pct
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

**事务隔离级别：**

```sql
-- PostgreSQL 支持 4 种隔离级别
SET TRANSACTION ISOLATION LEVEL
    READ COMMITTED      -- 默认，大多数场景够用
    -- READ COMMITTED    -- SELECT 看到提交后的数据
    -- REPEATABLE READ   -- SELECT 整个事务内看到一致快照
    -- SERIALIZABLE      -- 完全串行，最严格
;

-- READ COMMITTED vs REPEATABLE READ：
BEGIN;
SELECT age FROM users WHERE id = 1;  -- READ COMMITTED: 30
                                  -- REPEATABLE READ: 25（快照）
-- 另一个事务 UPDATE users SET age = 30 WHERE id = 1; COMMIT;
SELECT age FROM users WHERE id = 1;  -- READ COMMITTED: 30
                                   -- REPEATABLE READ: 25（仍快照）
COMMIT;
```

#### 真实面试题

**题目：PostgreSQL 的 MVCC 是什么？它是如何工作的？为什么需要 VACUUM？**

**满分答案：**

**MVCC 核心原理：**

PostgreSQL 的 MVCC（Multi-Version Concurrency Control）让每个事务看到数据库的一个"快照"：

1. **每行有两个隐藏列**：`xmin`（创建版本）和 `xmax`（删除版本）
2. **UPDATE 时**：旧行标记删除，新行插入（不是原地更新）
3. **事务可见性**：根据事务 ID 和行版本号判断是否可见

**MVCC 带来的好处：**
- **读写不互斥**：读者不阻塞写者，写者不阻塞读者
- **非阻塞快照读**：无需加锁，读取历史版本也可以

**VACUUM 为什么必须：**

```sql
-- PostgreSQL UPDATE/DELETE 不真正删除数据
UPDATE users SET age = 30 WHERE id = 1;
-- 旧版本 age=25 的行仍然存在，标记为"死亡元组"

-- VACUUM 清理死亡元组，释放磁盘空间
VACUUM users;
```

**表膨胀问题：**

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| 频繁 UPDATE/DELETE | 死亡元组堆积 | 定期 VACUUM |
| autovacuum 跟不上 | 磁盘占用持续增长 | 调参或手动 VACUUM |
| 长事务 | 旧版本无法清理 | 避免长事务，及时 COMMIT |

**生产建议：**
- `autovacuum_vacuum_cost_delay = 2ms`（默认 0ms，高频更新建议开启）
- 大批量 DELETE 后立即 VACUUM
- `VACUUM FULL` 仅在维护窗口使用（会锁表）

### 18.2.2 WAL 与数据持久化

#### 知识点详解

**Write-Ahead Logging (WAL)：**

```
写入流程：
1. 先写 WAL 日志（必须成功）
2. 再写数据文件（可异步）

崩溃恢复：
读取 WAL，重放日志，恢复到一致状态
```

```sql
-- WAL 级别配置
ALTER SYSTEM SET wal_level = replica;  -- default（支持流复制）
ALTER SYSTEM SET wal_level = logical;  -- 支持逻辑复制
ALTER SYSTEM SET wal_level = hot_standby;  -- 热备

-- 归档配置
ALTER SYSTEM SET archive_mode = on;
ALTER SYSTEM SET archive_command = 'cp %p /archive/%f';
```

**检查点（Checkpoint）：**

```sql
-- 手动触发检查点（促进 WAL 回收）
CHECKPOINT;

-- 检查点配置
ALTER SYSTEM SET checkpoint_timeout = '5min';
ALTER SYSTEM SET max_wal_size = '1GB';

-- 查看 WAL 状态
SELECT * FROM pg_wal_lsn_status();
SELECT pg_current_wal_lsn();  -- 当前 WAL 位置
```

#### 真实面试题

**题目：PostgreSQL 的 WAL 机制是什么？为什么先写日志再写数据？**

**满分答案：**

**WAL（预写日志）机制：**

PostgreSQL 所有数据变更先写入 WAL（预写日志），再异步写入数据文件。

**为什么先写日志：**

| 方式 | 优点 | 缺点 |
|------|------|------|
| 先写数据 | - | 崩溃可能数据不完整 |
| 先写 WAL | 崩溃可恢复，I/O 顺序写性能好 | 日志写入延迟 |

**核心优势：**

1. **崩溃恢复**：只需重放 WAL，比扫描整个数据文件快 100x
2. **热备份**：支持在线全量/增量备份
3. **流复制**：WAL 实时传输到从库，保证主从一致
4. **时间点恢复（PITR）**：WAL 归档 + 全量备份 = 任意时间点恢复

**I/O 优化原理：**

```sql
-- 随机写变顺序写：WAL 是追加写（顺序 I/O）
-- 数据文件是随机写（随机 I/O，慢）
-- WAL 顺序写入 → 批量合并 → 减少磁盘寻道
```

### 18.2.3 分区表（Partitioning）

#### 知识点详解

**范围分区（按时间）：**

```sql
-- 创建分区表
CREATE TABLE orders (
    id BIGSERIAL,
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2),
    created_at DATE NOT NULL
) PARTITION BY RANGE (created_at);

-- 创建按月分区的子表
CREATE TABLE orders_2026_01 PARTITION OF orders
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE orders_2026_02 PARTITION OF orders
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

CREATE TABLE orders_2026_03 PARTITION OF orders
    FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

-- 默认分区（捕获所有未匹配）
CREATE TABLE orders_default PARTITION OF orders DEFAULT;

-- 分区表上建索引
CREATE INDEX ON orders (user_id);  -- 自动在所有子表创建
CREATE INDEX ON orders (created_at, user_id);
```

**列表分区（按分类）：**

```sql
CREATE TABLE products (
    id SERIAL,
    category TEXT,
    name TEXT
) PARTITION BY LIST (category);

CREATE TABLE products_electronics PARTITION OF products
    FOR VALUES IN ('electronics', 'computers');
CREATE TABLE products_clothing PARTITION OF products
    FOR VALUES IN ('clothing', 'shoes');
```

**分区裁剪（Partition Pruning）：**

```sql
-- PostgreSQL 自动跳过不相关分区
EXPLAIN SELECT * FROM orders
WHERE created_at = '2026-03-15';
-- 只会扫描 orders_2026_03 分区，其他分区被裁剪

-- 启用/禁用裁剪
SET enable_partition_pruning = on;  -- 默认 on
```

#### 真实面试题

**题目：PostgreSQL 分区表有什么用？分区裁剪是什么？如何管理大量分区？**

**满分答案：**

**分区表的作用：**

将大表按维度拆分为多个物理子表，逻辑上还是一个表：

| 优势 | 说明 |
|------|------|
| 查询性能 | 只扫描相关分区，全表扫描变小 |

| 并行操作 | 不同分区可并行 vacuum/索引 |
| 数据管理 | 历史数据归档更简单 |

```sql
Gather  (cost=...)
  ->  Append on orders_2026_03  -- 其他分区被裁剪

*/

```

**分区最佳实践：**


### 18.2.4 视图与物化视图

#### 知识点详解

**普通视图（逻辑视图）：**

```sql
-- 创建视图
CREATE VIEW active_orders AS
SELECT o.id, o.amount, u.name, u.email
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'completed';

-- 使用视图
SELECT * FROM active_orders WHERE amount > 100;

-- 可更新的视图（满足条件时可 UPDATE/DELETE）
CREATE VIEW simple_users AS
SELECT id, name, email FROM users WHERE status = 'active'
WITH CHECK OPTION;  -- 强制所有 INSERT/UPDATE 满足 WHERE 条件

-- 修改视图
CREATE OR REPLACE VIEW active_orders AS
SELECT ...;  -- 替换原视图定义
```

**物化视图（物理视图）：**

```sql
-- 创建物化视图（实际存储数据）
CREATE MATERIALIZED VIEW monthly_sales AS
SELECT
    DATE_TRUNC('month', created_at) as month,
    COUNT(*) as order_count,
    SUM(amount) as total_amount
FROM orders
GROUP BY DATE_TRUNC('month', created_at);

-- 物化视图索引
CREATE INDEX ON monthly_sales (month);

-- 刷新数据
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales;
-- CONCURRENTLY: 不阻塞查询（需唯一索引）
-- 无 CONCURRENTLY: 排他锁，写入阻塞

-- 定时刷新（cron 或 pgAgent）
SELECT cron.schedule('refresh_monthly_sales', '0 2 * * *',
    'REFRESH MATERIALIZED VIEW monthly_sales');
```

**视图 vs 物化视图对比：**

| 特性 | 视图 | 物化视图 |
|------|------|---------|
| 存储数据 | ❌ 不存储 | ✅ 存储 |
| 查询速度 | 每次执行 SQL | 直接读（快 10-100x）|
| 数据新鲜度 | 实时 | 刷新时更新 |
| 磁盘占用 | 无 | 有 |
| 更新方式 | 自动 | 手动/定时 |

#### 真实面试题

**题目：PostgreSQL 的视图和物化视图有什么区别？分别在什么场景下使用？**

**满分答案：**

**核心区别：**

| 特性 | 视图 | 物化视图 |
|------|------|---------|
| 是否存储数据 | 否（实时查询） | 是（快照存储）|
| 查询速度 | 慢（执行底层 SQL）| 快（直接读）|
| 数据一致性 | 始终最新 | 刷新前可能过期 |
| 磁盘占用 | 无 | 有 |

**使用场景：**

**视图适用：**
- 简化复杂查询（频繁访问的 JOIN 结果）
- 隐藏表结构细节（提供只读接口）
- 行级/列级安全（限制不同角色看到不同数据）

**物化视图适用：**
- **报表/数据分析**：大表聚合，每天/每小时刷新
- **复杂计算**：含多表 JOIN + 聚合，避免每次重复计算
- **跨库查询**：定期拉取远程数据，本地缓存

```sql
-- 典型场景：电商月报表
CREATE MATERIALIZED VIEW monthly_report AS
SELECT
    u.region,
    DATE_TRUNC('month', o.created_at) as month,
    COUNT(DISTINCT u.id) as buyer_count,
    SUM(o.amount) as gmv
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'completed'
GROUP BY u.region, DATE_TRUNC('month', o.created_at);

-- 查询月报：毫秒级（vs 视图可能需要几十秒）
SELECT * FROM monthly_report WHERE month = '2026-03-01';
```

### 18.2.5 存储过程、函数与触发器

#### 知识点详解

**PL/pgSQL 函数：**

```sql
-- 基础函数
CREATE OR REPLACE FUNCTION get_user_count(status_arg TEXT)
RETURNS INTEGER AS $$
DECLARE
    count_val INTEGER;
BEGIN
    SELECT COUNT(*) INTO count_val
    FROM users
    WHERE status = status_arg;
    RETURN COALESCE(count_val, 0);
END;
$$ LANGUAGE plpgsql;

SELECT get_user_count('active');

-- 返回表
CREATE OR REPLACE FUNCTION get_orders_by_user(user_id_arg BIGINT)
RETURNS TABLE(id BIGINT, amount DECIMAL, status TEXT) AS $$
BEGIN
    RETURN QUERY
    SELECT o.id, o.amount, o.status
    FROM orders o
    WHERE o.user_id = user_id_arg;
END;
$$ LANGUAGE plpgsql;

-- 返回 SETOF 行
CREATE OR REPLACE FUNCTION get_top_users(limit_n INTEGER DEFAULT 10)
RETURNS SETOF users AS $$
BEGIN
    RETURN QUERY SELECT * FROM users ORDER BY created_at DESC LIMIT limit_n;
END;
$$ LANGUAGE plpgsql;

-- 异常处理
CREATE OR REPLACE FUNCTION create_order(...)
RETURNS BIGINT AS $$
DECLARE
    new_id BIGINT;
BEGIN
    INSERT INTO orders (...) VALUES (...) RETURNING id INTO new_id;
    RETURN new_id;
EXCEPTION
    WHEN unique_violation THEN
        RAISE EXCEPTION '订单号已存在';
    WHEN OTHERS THEN
        RAISE EXCEPTION '创建订单失败: %', SQLERRM;
END;
$$ LANGUAGE plpgsql;
```

**触发器：**

```sql
-- 触发器：自动记录变更历史
CREATE TABLE users_audit (
    id SERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    action TEXT NOT NULL,  -- 'INSERT'/'UPDATE'/'DELETE'
    old_data JSONB,
    new_data JSONB,
    changed_at TIMESTAMPTZ DEFAULT NOW(),
    changed_by TEXT DEFAULT current_user
);

CREATE OR REPLACE FUNCTION audit_users()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        INSERT INTO users_audit (user_id, action, new_data)
        VALUES (NEW.id, TG_OP, to_jsonb(NEW));
        RETURN NEW;
    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO users_audit (user_id, action, old_data, new_data)
        VALUES (OLD.id, TG_OP, to_jsonb(OLD), to_jsonb(NEW));
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        INSERT INTO users_audit (user_id, action, old_data)
        VALUES (OLD.id, TG_OP, to_jsonb(OLD));
        RETURN OLD;
    END IF;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER trg_audit_users
AFTER INSERT OR UPDATE OR DELETE ON users
FOR EACH ROW EXECUTE FUNCTION audit_users();

-- 行级 vs 语句级触发器
CREATE TRIGGER trg_check_salary
AFTER UPDATE OF salary ON employees
FOR EACH ROW
WHEN (OLD.salary IS DISTINCT FROM NEW.salary)
EXECUTE FUNCTION check_salary_change();
```

#### 真实面试题

**题目：PostgreSQL 的触发器有什么用？函数和存储过程有什么区别？**

**满分答案：**

**触发器应用场景：**

| 场景 | 触发器作用 |
|------|-----------|
| 审计日志 | 每次变更自动记录历史 |
| 数据验证 | INSERT/UPDATE 前检查约束 |
| 级联更新 | 主表更新，自动更新关联表 |
| 软删除 | DELETE 时改为 UPDATE status='deleted' |
| 自动统计 | 插入/删除后更新计数器表 |

**函数 vs 存储过程：**

| 特性 | 函数 | 存储过程 |
|------|------|---------|
| 返回值 | 必须有（RETURN）| 可选（OUT 参数）|
| 调用方式 | SELECT f() | CALL proc() |
| 事务控制 | 不能包含 COMMIT/ROLLBACK | 可以（事务内可分支）|
| 用途 | 计算、转换（SELECT 中用）| 批量操作、ETL |

### 18.2.7 pgvector 向量数据库

#### 知识点详解

**向量数据库是 AI/RAG 应用的核心：**

```sql
-- 安装扩展
CREATE EXTENSION vector;

-- 创建向量列
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1536)  -- OpenAI text-embedding-3-small 1536维
);

-- 插入向量（通常由 AI 模型生成）
INSERT INTO documents (content, embedding)
VALUES ('什么是PostgreSQL',
    '[0.1, 0.2, -0.3, ...]::real[]');  -- 实际由模型生成

-- 余弦相似度搜索（RAG 场景）
SELECT content, (embedding <=> $1::vector) as similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 5;

-- <#>: 内积（Inner Product，适合归一化向量）
-- <->: 欧氏距离（L2）
-- <=>: 余弦距离（Cosine，推荐）
```

**向量索引（IVFFlat / HNSW）：**

```sql
-- IVF-Flat 索引（适合百万级向量）
CREATE INDEX ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- HNSW 索引（更快，更耗内存，适合千万级）
CREATE INDEX ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- 查询时设置搜索范围
SET hnsw.ef_search = 50;  -- 越高越精确，越慢
```

**RAG 完整示例：**

```sql
-- 1. 文档入库
INSERT INTO documents (content, embedding)
VALUES ($1, $2);  -- $1=文档内容，$2=embedding

-- 2. 相似度检索
SELECT content FROM documents
ORDER BY embedding <=> $query_embedding
LIMIT 3;

-- 3. 上下文构建
SELECT string_agg(content, ' ') as context
FROM (
    SELECT content FROM documents
    ORDER BY embedding <=> $query_embedding
    LIMIT 5
) top_docs;
```

#### 真实面试题

**题目：PostgreSQL 如何支持向量搜索？pgvector 有什么用？**

**满分答案：**

**为什么需要向量搜索：**

传统数据库只能精确匹配，向量数据库能搜索"相似的内容"：

```
用户问："PostgreSQL怎么建索引？"
        ↓ （embedding 模型）
向量 [0.12, -0.45, 0.78, ...]
        ↓ （向量相似度搜索）
找到："PostgreSQL建索引的方法有..." （内容相似）
```

**pgvector 使用：**

```sql
-- 存入文档向量
CREATE TABLE docs (id SERIAL, content TEXT, embedding VECTOR(1536));
CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops);

-- 检索（最相似的内容）
SELECT content FROM docs
ORDER BY embedding <=> $user_query_embedding
LIMIT 3;
```

**向量搜索在 AI 前端的应用：**

| 步骤 | 说明 |
|------|------|
| 1. 文档分块 | 用户上传文档 → 前端/后端切分 |
| 2. 向量化 | 调用 Embedding API 转向量 |
| 3. 存入 PG | PG 保存向量 + 原文 |
| 4. 检索 | 用户提问 → 转向量 → 相似搜索 |
| 5. 增强生成 | 将 Top-K 文档注入 Prompt |

**pgvector vs 专业向量数据库：**

| 对比 | pgvector | Milvus / Pinecone |
|------|---------|------------------|
| 规模 | 千万级 | 亿级 |
| 成熟度 | 较新 | 成熟 |
| 优势 | 简单、事务支持 | 超大规模、专属优化 |

### 18.2.8 连接池与性能调优

#### 知识点详解

**PostgreSQL 连接管理：**

```sql
-- 最大连接数（默认 100）
ALTER SYSTEM SET max_connections = 200;

-- 查看当前连接
SELECT count(*), state FROM pg_stat_activity GROUP BY state;

-- 杀掉空闲连接
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle' AND state_change < NOW() - INTERVAL '1 hour';

-- 连接池推荐：PgBouncer
-- /etc/pgbouncer/pgbouncer.ini
; [databases]
; mydb = host=127.0.0.1 port=5432 dbname=mydb

; [pgbouncer]
; pool_mode = transaction  -- 事务模式（推荐）
listen_port = 6432
listen_addr = 127.0.0.1
max_client_conn = 1000
default_pool_size = 20
```

**性能调优参数：**

```sql
-- shared_buffers（建议 OS 内存的 25%）
ALTER SYSTEM SET shared_buffers = '4GB';

-- work_mem（排序/哈希内存，单位连接）
ALTER SYSTEM SET work_mem = '64MB';

-- maintenance_work_mem（VACUUM/索引构建内存）
ALTER SYSTEM SET maintenance_work_mem = '512MB';

-- effective_cache_size（查询规划器用，不分配内存）
ALTER SYSTEM SET effective_cache_size = '12GB';

-- random_page_cost（SSD 设为 1.1，HDD 设为 4.0）
ALTER SYSTEM SET random_page_cost = 1.1;

-- 时区
ALTER SYSTEM SET timezone = 'Asia/Shanghai';
```

**慢查询分析与优化：**

```sql
-- 开启慢查询日志
ALTER SYSTEM SET log_min_duration_statement = 1000;  -- 超过1秒记录

-- 查看慢查询
SELECT
    query,
    calls,
    round(mean_exec_time::numeric, 2) as avg_ms,
    round(total_exec_time::numeric, 2) as total_ms,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;

-- 查看实时占用
SELECT
    pid,
    usename,
    application_name,
    state,
    round(EXTRACT(EPOCH FROM (NOW() - state_change)) / 60, 2) as idle_minutes,
    query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY state_change;
```

**EXPLAIN ANALYZE 分析：**

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2026-01-01'
GROUP BY u.id;

-- 关键指标：
-- Seq Scan: 全表扫描（需优化）
-- Index Scan: 索引扫描（好）
-- Bitmap Heap Scan: 位图索引扫描（大表）
-- Nested Loop / Hash Join / Merge Join: 连接方式
-- rows=预估行数, actual rows=实际行数（差异大说明统计信息过时）
```

### 18.2.9 主从复制与高可用

#### 知识点详解

**流复制（Streaming Replication）：**

```sql
-- 主库：创建复制用户
CREATE USER repl WITH REPLICATION PASSWORD 'password';

-- 主库：修改 pg_hba.conf
host replication repl 0.0.0.0/0 md5

-- 主库：创建复制槽（保证 WAL 不丢失）
SELECT pg_create_physical_replication_slot('replica1_slot');

-- 主库：pg.conf
ALTER SYSTEM SET wal_level = replica;
ALTER SYSTEM SET max_wal_senders = 10;
ALTER SYSTEM SET wal_keep_size = '1GB';

-- 从库配置（pg_basebackup）
pg_basebackup -h master_host -U repl -D /var/lib/postgresql/data -P -Xs -R

-- 从库：recovery.conf（PG 12+ 用 postgresql.conf）
primary_conninfo = 'host=master_host port=5432 user=repl application_name=replica1'
```

**逻辑复制：**

```sql
-- 逻辑复制（发布-订阅模型）
-- 主库：创建发布
CREATE PUBLICATION mypub FOR TABLE users, orders;

-- 从库：创建订阅
CREATE SUBSCRIPTION mysub CONNECTION 'host=master port=5432'
    PUBLICATION mypub;

-- 复制特定行/列
CREATE PUBLICATION mypub FOR TABLE orders
    WHERE (status = 'completed');
```

**高可用方案：**

| 方案 | 原理 | 切换时间 | 复杂度 |
|------|------|---------|--------|
| PgBouncer + keepalived | 连接池+VIP | 秒级 | 低 |
| Patroni + etcd | 分布式协调 | <30s | 中 |
| Pgpool-II | 负载均衡+健康检查 | <30s | 中 |
| 云原生 | Aurora / RDS | 自动 <1s | 低 |
| Stolon | Consul/etcd | <30s | 中 |

```bash
# Patroni 架构（生产推荐）
# etcd: 存储集群状态
# Patroni: 管理 PostgreSQL 实例
# HAProxy: 路由流量到主库

# 典型配置 patroni.yml:
# postgresql:
#   parameters:
#     wal_level: replica

#   auth: 'postgres:password'

```

### 18.2.10 数据库安全

#### 知识点详解

**权限管理：**

```sql
-- 角色与用户
CREATE ROLE readonly;           -- 创建角色
CREATE ROLE developer LOGIN;    -- 可登录用户
CREATE ROLE admin CREATEROLE;   -- 可创建其他角色

-- 权限授予
GRANT SELECT, INSERT ON ALL TABLES IN SCHEMA public TO readonly;
GRANT USAGE ON SCHEMA public TO readonly;

-- 行级安全（RLS）
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
CREATE POLICY orders_own_policy ON orders
    FOR ALL
    USING (user_id = current_user_id());  -- 用户只能看自己的订单

-- 列级安全（部分字段脱敏）
CREATE VIEW users_public AS
SELECT id, name, email,
    CASE WHEN current_role = 'admin' THEN phone ELSE NULL END as phone
FROM users;

-- 数据加密
-- 透明加密（pg_stat_statements 等扩展）
CREATE EXTENSION pgcrypto;

-- 应用层加密（密码等敏感数据）
UPDATE users SET password = crypt('plaintext', gen_salt('bf'))
WHERE id = 1;
```

#### 真实面试题

**题目：PostgreSQL 如何实现行级安全和列级脱敏？**

**满分答案：**

**行级安全（RLS）：**

```sql
-- 场景：多租户系统，用户只能看自己的数据
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;  -- 强制（即使管理员也受约束）

CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_setting('app.current_tenant')::BIGINT);

-- 应用连接时设置租户
SET app.current_tenant = '123';
SELECT * FROM orders;  -- 自动过滤，只返回 tenant_id=123 的数据
```

**列级脱敏：**

```sql
-- 场景：普通用户看不到手机号，管理员可见

SELECT
    id, name, email,
    CASE
        WHEN current_user = 'admin' THEN phone
        ELSE NULL
    END as phone
FROM users;

-- 或用 SECURITY DEFINER 函数
CREATE FUNCTION get_user_phone(user_id BIGINT)
RETURNS TEXT SECURITY DEFINER
AS $$
BEGIN
    RETURN (SELECT phone FROM users WHERE id = user_id);
```

## 18.3 MySQL 对比专题

### 18.3.1 MySQL vs PostgreSQL 核心差异

#### 知识点详解

**架构对比：**

| 维度 | MySQL | PostgreSQL |
|------|-------|-----------|
| 引擎 | InnoDB/MyISAM/Memory | 统一（PostgreSQL 原生）|
| 并发控制 | MVCC（InnoDB）+ 表锁 | MVCC + 行锁（完整）|
| SQL 标准 | 部分兼容 | 高度兼容 |
| 扩展性 | 插件少 | 通过扩展可扩展 |
| JSON 支持 | 5.7+ JSON 类型（性能一般）| JSONB 高性能 |
| 全文搜索 | 内置（InnoDB/MyISAM）| 内置（更强大）|
| 地理信息 | 有限 | PostGIS（顶级）|
| 主从复制 | binlog | WAL（更可靠）|
| 连接数 | 轻量（默认 151）| 重量（可支持更多）|

**InnoDB vs PostgreSQL MVCC：**

```sql
-- MySQL InnoDB MVCC：
-- 依赖隐藏列（DB_TRX_ID, DB_ROLL_PTR）+ Undo Log
-- 隔离级别：READ UNCOMMITTED / READ COMMITTED / REPEATABLE READ / SERIALIZABLE
-- 默认 REPEATABLE READ（快照读）

-- PostgreSQL MVCC：
-- 每行 xmin/xmax + 事务快照
-- 隔离级别：READ COMMITTED / REPEATABLE READ / SERIALIZABLE
-- 默认 READ COMMITTED（最常用）
```

**分页对比：**

```sql
-- MySQL: LIMIT offset, count（大数据 offset 性能差）
SELECT * FROM orders ORDER BY id LIMIT 1000000, 20;
-- offset 越大越慢（需扫描1000020行返回20行）

-- PostgreSQL: cursor 分页（推荐）
SELECT * FROM orders
WHERE id > 1000000  -- 上一页最后一条的 id
ORDER BY id
LIMIT 20;
-- 无论翻到第几页，性能稳定
```

#### 真实面试题

**题目：MySQL 和 PostgreSQL 有什么区别？如何选择？**

**核心区别：**

|------|-------|-----------|

**如何选择：**

| 场景 | 推荐 |

### 18.3.2 MySQL 索引与 InnoDB 机制

#### 知识点详解

**InnoDB 特性：**

```sql
-- InnoDB 聚簇索引（主键索引即数据）
-- 数据按主键顺序存储，非主键查询需回表

-- MySQL 组合索引最左前缀
CREATE INDEX idx_order ON orders (user_id, status, created_at);
-- 可用于：
-- WHERE user_id = 1
-- WHERE user_id = 1 AND status = 'paid'
-- WHERE user_id = 1 AND status = 'paid' AND created_at > '2026-01-01'
-- 不可用于：
-- WHERE status = 'paid'
-- WHERE created_at > '2026-01-01'

-- MySQL 覆盖索引
CREATE INDEX idx_cover ON orders (user_id, status, amount);
-- SELECT status, amount FROM orders WHERE user_id = 1;（不回表）
```

**MySQL vs PostgreSQL 索引对比：**

| 特性 | MySQL | PostgreSQL |
|------|-------|-----------|
| 默认索引 | B-Tree | B-Tree |
| Hash 索引 | 支持 | 支持 |
| 全文索引 | 支持 | 支持（更强）|
| 组合索引 | 支持 | 支持 |
| 部分索引 | ❌ | ✅ |
| 表达式索引 | ❌ | ✅ |
| JSON 索引 | 有限 | GIN 索引 |
| GiST/GIN | ❌ | ✅ |

#### 真实面试题

**题目：MySQL 组合索引的最左前缀原则是什么？为什么 PostgreSQL 的部分索引很有用？**

**满分答案：**

**MySQL 最左前缀原则：**

```sql
CREATE INDEX idx ON orders (user_id, status, created_at);

-- ✅ 可用索引
SELECT * FROM orders WHERE user_id = 1;
SELECT * FROM orders WHERE user_id = 1 AND status = 'paid';
SELECT * FROM orders WHERE user_id = 1 AND status = 'paid' AND created_at > '2026-01-01';

-- ❌ 无法使用索引（跳过最左列）
SELECT * FROM orders WHERE status = 'paid';
SELECT * FROM orders WHERE created_at > '2026-01-01';

-- ⚠️ 索引列参与计算/函数时失效
SELECT * FROM orders WHERE YEAR(created_at) = 2026;  -- ❌
SELECT * FROM orders WHERE created_at >= '2026-01-01';  -- ✅
```

**PostgreSQL 部分索引的价值：**

```sql
-- 场景：只查询"活跃用户"，且活跃用户只占 5%
-- 普通索引：索引 95% 的无用数据
CREATE INDEX idx_inactive ON users (last_login)
WHERE status = 'inactive';  -- 只索引 inactive 用户
-- 索引体积小 95%，查询快 10x

-- 场景：软删除表
CREATE INDEX idx_orders_active ON orders (created_at)
WHERE deleted_at IS NULL;  -- 只索引未删除订单
SELECT * FROM orders WHERE created_at > '2026-01-01' AND deleted_at IS NULL;
-- 比索引 deleted_at IS NOT NULL 高效得多
```

## 18.4 MongoDB 对比专题

### 18.4.1 MongoDB 特点与适用场景

#### 知识点详解

**文档模型 vs 关系模型：**

```javascript
// MongoDB 文档（JSON/BSON）
{
    "_id": ObjectId("..."),
    "name": "Alice",
    "email": "alice@example.com",
    "address": {
        "city": "Shanghai",
        "district": "Pudong"
    },
    "orders": [
        { "id": 1, "amount": 99, "status": "completed" },
        { "id": 2, "amount": 199, "status": "pending" }
    ]
}

// PostgreSQL 等价（需多表 JOIN）
-- users 表 + orders 表 + JSONB 列（混合方案）
```

**核心操作：**

```javascript
// 插入
db.users.insertOne({ name: "Alice", email: "alice@example.com" });
db.users.insertMany([...]);

// 查询
db.users.find({ name: "Alice" });                        // 等值
db.users.find({ "address.city": "Shanghai" });           // 嵌套
db.users.find({ orders: { $elemMatch: { amount: { $gt: 100 } } } });  // 数组查询
db.users.find({}, { name: 1, email: 1, _id: 0 });        // 投影
db.users.find().sort({ created_at: -1 }).skip(20).limit(10);  // 分页

// 聚合管道（类比 SQL）
db.orders.aggregate([
    { $match: { status: "completed" } },          // WHERE
    { $group: { _id: "$user_id", total: { $sum: "$amount" } } },  // GROUP BY
    { $sort: { total: -1 } },                     // ORDER BY
    { $limit: 10 }                               // LIMIT
]);

// 更新
db.users.updateOne(
    { _id: 1 },
    { $set: { name: "Alice Wang" }, $inc: { age: 1 } }
);

// 索引
db.users.createIndex({ email: 1 }, { unique: true });
db.users.createIndex({ name: "text", bio: "text" });  // 全文索引
```

**MongoDB 适用场景：**

| 场景 | 原因 |
|------|------|
| 快速原型/敏捷开发 | 灵活 schema，随时改结构 |
| 内容管理/CMS | 文档自然映射内容对象 |
| 物联网（IoT）| 时间序列数据存储 |
| 实时分析 | 高写入性能 + 聚合管道 |
| 日志系统 | 大量写入 + 灵活字段 |

| 文件元数据 | PostgreSQL + S3 | 可靠存储 |
| 缓存 | Redis | 毫秒级访问 |

**实践原则：**
- 优先 PostgreSQL（一个数据库解决大部分问题）
- 需要水平扩展时才考虑 MongoDB
- 不要为了"时尚"选 MongoDB（运维复杂度高）
- **PostgreSQL 16+ 已经是"超级数据库"**，很多场景无需额外数据库

### 18.5.2 SQL 与 NoSQL 面经

#### 真实面试题

**题目：为什么 NoSQL 数据库近年来很火？SQL 会被淘汰吗？**

**满分答案：**

**NoSQL 兴起的背景：**

1. **海量数据**：Web 2.0 时代数据量爆炸式增长，传统 DB 无法水平扩展
2. **敏捷开发**：schema 固定阻碍快速迭代
3. **高并发写入**：社交网络需要百万级写入/秒
4. **不同数据结构**：图谱、文档、时序——关系模型不自然

**SQL 不会淘汰的原因：**

| NoSQL 优势 | SQL 弥补 |
|-----------|---------|
| 水平扩展 | PostgreSQL 分区 + Citus = 水平扩展 |
| 灵活 schema | PostgreSQL JSONB = schema-less |
| 高写入 | PostgreSQL 15 WAL 优化，高并发 |
| 复杂查询 | MongoDB Aggregation 无法匹敌 SQL |

**结论：**
> NoSQL 擅长特定场景（海量写入、灵活文档），但 PostgreSQL 16+ 已经是一个"全能数据库"。对于大多数项目，**PostgreSQL 优先**，只有在 NoSQL 有明确优势时才考虑。

**题目：Redis 和 PostgreSQL 缓存如何配合？**

**满分答案：**

**缓存策略：**

```
请求 → Redis（命中缓存）→ 返回
           ↓（未命中）
       PostgreSQL → 查询 → 写入Redis → 返回
```

```typescript
// 应用层缓存
async function getUser(id: number) {
    const cacheKey = `user:${id}`;
    const cached = await redis.get(cacheKey);
    if (cached) return JSON.parse(cached);

    const user = await db.query('SELECT * FROM users WHERE id = $1', [id]);
    await redis.setex(cacheKey, 3600, JSON.stringify(user));  // 1小时 TTL
    return user;
}
```

**缓存失效策略：**

| 策略 | 实现 | 适用场景 |
|------|------|---------|
| Cache Aside | 读时写缓存，写时删缓存 | 最常用 |
| Write Through | 写缓存+写DB同步 | 数据一致性要求高 |
| Write Behind | 写缓存，异步写DB | 写入性能高，有数据丢失风险 |

**缓存问题：**

| 问题 | 解决方案 |
|------|---------|
| 缓存穿透 | 布隆过滤器 / 空值缓存 |
| 缓存击穿 | 互斥锁 / 永不过期+异步更新 |
| 缓存雪崩 | 随机 TTL / Redis Cluster |

**题目：PostgreSQL 中，如何优化大表的 JOIN 性能？**

**满分答案：**

**优化步骤：**

```sql
-- 1. 分析执行计划
EXPLAIN (ANALYZE, BUFFERS)
SELECT u.name, o.amount
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2026-01-01';

-- 2. 确保 JOIN 键有索引
CREATE INDEX idx_orders_user ON orders (user_id);
CREATE INDEX idx_users_created ON users (created_at);

-- 3. 减少参与 JOIN 的数据量
-- ❌ SELECT * FROM users JOIN orders...
-- ✅ SELECT u.name, o.amount FROM users...（只取需要的列）

-- 4. 分解大 JOIN（分批处理）
WITH user_batch AS (
    SELECT id, name FROM users WHERE created_at > '2026-01-01' LIMIT 1000
)
SELECT ub.name, o.amount
FROM user_batch ub
LEFT JOIN orders o ON ub.id = o.user_id;

-- 5. 用 CTE 缓存中间结果
WITH active_users AS (
    SELECT id FROM users WHERE status = 'active'  -- 先过滤小结果集
)
SELECT ... FROM active_users u JOIN orders o ON u.id = o.user_id;
```

**JOIN 类型选择：**

| 场景 | 推荐 JOIN |
|------|---------|
| 小表 JOIN 大表 | Hash Join（大表 Build，小表 Probe）|
| 有序数据 JOIN | Merge Join |
| 小结果集 JOIN | Nested Loop |

**题目：什么是数据库的 ACID 特性？PostgreSQL 如何保证？**

**满分答案：**

**ACID 四个特性：**

| 特性 | 说明 | PostgreSQL 实现 |
|------|------|---------------|
| Atomic（原子性）| 事务要么全成功，要么全失败 | WAL + 回滚段 |
| Consistency（一致性）| 事务前后，数据满足所有约束 | 约束检查（PK/FK/CHECK）|
| Isolation（隔离性）| 并发事务互不干扰 | MVCC + 锁 |
| Durability（持久性）| 提交的数据不丢失 | WAL + 落盘 |

**PostgreSQL 事务示例：**

```sql
BEGIN;
    INSERT INTO orders (user_id, amount) VALUES (1, 99.9);
    UPDATE accounts SET balance = balance - 99.9 WHERE user_id = 1;
    -- 两步必须同时成功或失败
COMMIT;  -- WAL 写入 → 落盘 → 返回成功
-- 如果此时机器宕机，重启后重放 WAL，恢复到一致状态
```

**ACID vs BASE（NoSQL）：**

| 模型 | 特性 | 代表数据库 |
|------|------|---------|
| ACID | 强一致性 | PostgreSQL, MySQL |
| BASE | 最终一致性 | MongoDB, Cassandra |
| CAP | 一致性/可用性/分区容忍只能同时满足 2 个 | |

**题目：PostgreSQL 中 CTE（WITH）的作用？递归 CTE 有什么用？**

**满分答案：**

**CTE 作用：**

```sql
-- 1. 简化复杂查询（可读性）
WITH
    recent_orders AS (
        SELECT user_id, SUM(amount) as total
        FROM orders
        WHERE created_at > '2026-01-01'
        GROUP BY user_id
    )
SELECT u.name, r.total
FROM users u
JOIN recent_orders r ON u.id = r.user_id
WHERE r.total > 1000;
-- 相比嵌套子查询，CTE 可读性更强

-- 2. 多次引用中间结果
WITH daily AS (
    SELECT DATE_TRUNC('day', created_at) as day, SUM(amount) as total
    FROM orders GROUP BY day
)
SELECT
    d1.day,
    d1.total as today,
    d1.total - COALESCE(d2.total, 0) as growth
FROM daily d1
LEFT JOIN daily d2 ON d2.day = d1.day - INTERVAL '1 day';  -- 引用两次
```

**递归 CTE 应用：**

```sql
-- 场景：组织架构树
WITH RECURSIVE org AS (
    -- 递归起点：CEO（没有上级的人）
    SELECT id, name, manager_id, 1 as level, ARRAY[name] as path
    FROM employees WHERE manager_id IS NULL

    UNION ALL

    -- 递归：经理的下属
    SELECT e.id, e.name, e.manager_id, o.level + 1, o.path || e.name
    FROM employees e
    JOIN org o ON e.manager_id = o.id
)
SELECT * FROM org ORDER BY level, name;

/* 输出：
id | name  | level | path
---|-------|-------|----------
1  | 张总  |  1    | {张总}
2  | 李总  |  2    | {张总,李总}
5  | Alice |  3    | {张总,李总,Alice}
*/
```

**递归 CTE 场景总结：**

| 场景 | 说明 |
|------|------|
| 组织架构 | 员工→经理→总监→CTO |
| 树形评论 | 评论→回复→回复的回复 |
| 路径查找 | 地铁线路、导航路径 |
| 图遍历 | 社交网络好友推荐 |
